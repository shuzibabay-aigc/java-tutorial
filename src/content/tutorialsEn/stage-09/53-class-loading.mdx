# Class Loading

The previous chapter covered GC — how objects on the heap "die." This chapter covers how classes are "born" — how the JVM loads `.class` files from disk into memory and turns them into usable classes. This is the **Class Loading Mechanism**.

Class loading is Java's "customs" — `.class` files are like imported goods, going through "declaration (loading) → inspection (verification) → valuation (preparation) → dispatch (resolution) → shelving (initialization)" — five checkpoints before they can be used. This chapter examines these five checkpoints, plus "parent delegation," the most-asked mechanism in interviews.

## 1. The Class Loading Process: Five Phases

The JVM spec defines class loading as five phases: **loading, verification, preparation, resolution, initialization**. Usage and unloading aren't strictly "loading," but the full lifecycle has seven phases:

```
loading → verification → preparation → resolution → initialization → usage → unloading
 └──────────linking───────────┘
```

Verification, preparation, and resolution together form **Linking**.

### 1.1 Loading

**Loading** is "finding the bytecode and reading it into memory" — three things:

1. **Obtain the binary byte stream defining this class via its fully qualified name** — from zip/jar, network, dynamic generation, file system, etc.
2. **Convert this byte stream into a method-area runtime data structure** — stored in metaspace.
3. **Generate a `Class` object on the heap** — as the access entry to the method-area data.

The "binary byte stream" has diverse sources:

- **jar/war** — most common.
- **Network** — the Applet era.
- **Dynamic generation** — CGLIB, ByteBuddy, dynamic proxies.
- **JSP compilation** — Tomcat compiles JSPs into classes.
- **Database** — some middleware.

Loading is the most developer-controllable phase — through **custom class loaders** you control the byte-stream source (covered later).

### 1.2 Verification

**Verification** ensures the bytecode is safe and legal — preventing malicious bytecode from crashing the JVM. Four categories:

- **File format** — magic number `0xCAFEBABE`, version, constant-pool indices valid.
- **Metadata** — class has a parent, doesn't extend a final class, field/method signatures legal.
- **Bytecode** — method body logic legal, operand stack doesn't overflow, jump instructions legal.
- **Symbolic references** — referenced classes, fields, methods actually exist, access permissions legal.

A failed verification throws `VerifyError`.

### 1.3 Preparation

**Preparation** allocates memory for **class variables (static)** and sets **zero values** — not initialized values!

```java
public static int x = 123;
// Preparation phase: x = 0  (zero value)
// Initialization phase: x = 123 (assigned)
```

Exception — `static final` constants known **at compile time** are assigned directly in the preparation phase:

```java
public static final int X = 123;   // Preparation phase: X = 123 (ConstantValue attribute)
public static final String S = "hello";   // Preparation phase: S = "hello"
```

Note that preparation handles only **class variables**, not **instance variables** — instance variables are allocated on the heap with the object, with zero values set at object creation (previous chapter).

### 1.4 Resolution

**Resolution** replaces **symbolic references** in the constant pool with **direct references**.

- **Symbolic reference** — a string describing the referenced target (e.g., `Ljava/lang/String;`, `java/lang/Object.hashCode:()I`). Independent of the target class's actual memory layout.
- **Direct reference** — a pointer, handle, or offset pointing to the target. Tied to the JVM memory layout.

Resolution can be **lazy** — some JVMs resolve a symbolic reference only on first use (lazy resolution), not necessarily at class-loading time.

### 1.5 Initialization

**Initialization** executes the class's `＜clinit＞` method — auto-generated by the compiler, merging **all static variable assignments** and **static blocks** in source order:

```java
public class Foo {
    static int a = 1;            // compiled: a = 1
    static { a = 2; b = 3; }     // compiled: a = 2; b = 3
    static int b = 4;            // compiled: b = 4 (overwrites the 3 above)
    // <clinit> content:
    // a = 1; a = 2; b = 3; b = 4;
}
```

`＜clinit＞` characteristics:

- **JVM guarantees thread safety** — multiple threads triggering initialization simultaneously have `＜clinit＞` execute once; others block. This is why the "static inner class singleton" implementation is thread-safe.
- **Parent class initializes first** — before a subclass's `＜clinit＞`, the parent's `＜clinit＞` runs.
- **Interface `＜clinit＞` doesn't require parent-interface initialization first** — interface initialization is independent.
- **If there are no static variable assignments/static blocks, the compiler doesn't generate `＜clinit＞`**.

### 1.6 When Initialization Triggers

**Active references** (must initialize):

1. The four bytecode instructions `new`, `getstatic`, `putstatic`, `invokestatic` — new object, read/write static field (non-final), call static method.
2. Reflection (`Class.forName`).
3. When initializing a subclass, the parent class initializes first.
4. The main class at JVM startup (the class with `main`).
5. The class corresponding to a `MethodHandle` handle.

**Passive references** (no initialization):

1. Accessing a parent's static field through a subclass — only the parent initializes, not the subclass.
2. `ClassName[] arr = new ClassName[10]` — doesn't initialize `ClassName`, only the array type.
3. Accessing `static final` constants (compile-time constants) — goes directly into the caller's constant pool; doesn't trigger the defining class's initialization.

```java
class Parent { static int x = 1; static { System.out.println("Parent init"); } }
class Child extends Parent { static { System.out.println("Child init"); } }

System.out.println(Child.x);   // only outputs "Parent init" + "1", not "Child init"
```

## 2. The Parent Delegation Model

### 2.1 Class Loader Hierarchy

The JVM has three built-in class loaders (JDK 9+ slightly adjusted):

| Loader | JDK 8 | JDK 9+ | What it loads |
| --- | --- | --- | --- |
| Bootstrap ClassLoader | C++ | C++ | `JAVA_HOME/lib` (rt.jar, java.lang.*) |
| Extension ClassLoader | `ExtClassLoader` | `PlatformClassLoader` | `JAVA_HOME/lib/ext` or JDK system modules |
| Application ClassLoader | `AppClassLoader` | `AppClassLoader` | classpath (app itself + deps) |

Hierarchy:

```
Bootstrap ClassLoader  (C++, no parent loader)
       ↑ parent
Extension/Platform ClassLoader
       ↑ parent
Application ClassLoader
       ↑ parent
User custom ClassLoaders
```

Note — "parent loader" isn't "parent class" (inheritance), but a **composition relationship** — each ClassLoader has a `parent` field pointing to its parent loader.

### 2.2 How Parent Delegation Works

When a class loader receives a load request, it **first delegates to the parent loader**; only if the parent fails does it load itself:

```java
// Simplified ClassLoader.loadClass logic
protected Class<?> loadClass(String name, boolean resolve) {
    // 1. Check if already loaded
    Class<?> c = findLoadedClass(name);
    if (c == null) {
        try {
            // 2. Delegate to parent
            if (parent != null) {
                c = parent.loadClass(name, false);
            } else {
                c = findBootstrapClassOrNull(name);
            }
        } catch (ClassNotFoundException e) {
            // parent can't find it
        }
        if (c == null) {
            // 3. Parents all failed, find it myself
            c = findClass(name);
        }
    }
    return c;
}
```

### 2.3 Why Parent Delegation

**Guarantees class global uniqueness** — the same class is loaded only once (by the highest-priority loader).

For example — `java.lang.Object` is always loaded by Bootstrap. If a user writes a `java.lang.Object` to override, it delegates to Bootstrap first, Bootstrap finds JDK's `Object` and loads it — the user's "fake" version is ignored. This is **security** — preventing core class replacement and duplicate class loading.

### 2.4 Class Uniqueness

The JVM identifies a class by **(fully qualified class name + class loader)**. The same class file loaded by two different ClassLoaders yields **two different Class objects** — `equals` returns `false`, `instanceof` doesn't match. This is the key to container isolation like Tomcat's.

## 3. Breaking Parent Delegation

Parent delegation is great, but some scenarios must "break" it — letting child loaders load first, or bypassing the parent.

### 3.1 Tomcat: Web App Isolation

Tomcat deploys multiple web apps, each with its own `WEB-INF/classes` and `WEB-INF/lib`. Requirements:

- **App A and App B's classes are mutually invisible** — avoid conflicts (e.g., A uses Spring 5, B uses Spring 4).
- **App classes can't be loaded by Tomcat itself** — otherwise, you'd need to restart Tomcat to update the app.
- **Tomcat's own classes are visible to apps** — apps need the Servlet API.

Tomcat designed a multi-layer ClassLoader:

```
Bootstrap
  ↑
Extension
  ↑
Application (System)
  ↑
Common (Tomcat common, loads $CATALINA_HOME/lib)
  ↑
┌──────────┴──────────┐
Catalina              Shared (optional)
(Tomcat internal classes)  ↑
                  ┌────┴────┐
              WebApp1     WebApp2
              (each app independent)
```

Each WebApp ClassLoader **breaks parent delegation** — it first looks in its own `WEB-INF/classes`, and only delegates to the parent if not found. This is "app-class-first" — same-named classes exist independently in each WebApp.

### 3.2 SPI: JDBC's Reverse Loading

**SPI (Service Provider Interface)** is Java's extension mechanism. Problem — `java.sql.DriverManager` (JDK internal) needs to load `com.mysql.cj.jdbc.Driver` (a third-party jar), but the Bootstrap ClassLoader can't see third-party jars on the classpath.

**Solution**: `Thread.currentThread().getContextClassLoader()` — the Thread Context ClassLoader. `DriverManager` uses the thread context ClassLoader (default AppClassLoader) to reverse-load the driver class, breaking the "parent loader can't see classes loaded by child loaders" limitation.

```java
// Inside DriverManager (simplified)
ServiceLoader<Driver> loaders = ServiceLoader.load(
    Driver.class,
    Thread.currentThread().getContextClassLoader()   // use thread context loader
);
```

This is a "parent loader asks child loader to load a class" reverse operation — a "loophole" in parent delegation, but necessary.

### 3.3 Hot Deployment: JRebel, Hot Reload

Hot deployment requires "modifying a class file takes effect without restarting the JVM." **Core idea** — each modification creates a new ClassLoader to reload the class. The new class and old class are different Class objects (different loaders), not interfering with each other. Once the old ClassLoader loses references, it's GC'd, and the old class is unloaded.

This is an extreme use of "breaking parent delegation" — the same class loaded multiple times, each time a "new version."

### 3.4 OSGi: Modular Class Loading

OSGi uses a "mesh" class-loading structure — each bundle has its own ClassLoader, bundles declare dependencies and are mutually visible. Fully breaking the hierarchy, supporting modularization. Java 9's JPMS is in some sense a simplified version of OSGi's ideas.

## 4. Custom Class Loaders

### 4.1 Why Customize

- **Load from non-standard sources** — network, encrypted files, database, in-memory generation.
- **Isolation** — same code, different versions coexisting.
- **Encryption protection** — encrypted class files, custom loader decrypts.
- **Hot deployment** — reload classes.

### 4.2 Implementation

Extend `ClassLoader`, override `findClass` (don't override `loadClass` — it would break parent delegation):

```java
public class MyClassLoader extends ClassLoader {
    private final String classPath;

    public MyClassLoader(String classPath, ClassLoader parent) {
        super(parent);
        this.classPath = classPath;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] bytes = loadClassBytes(name);
        if (bytes == null) throw new ClassNotFoundException(name);
        // defineClass turns bytecode into a Class
        return defineClass(name, bytes, 0, bytes.length);
    }

    private byte[] loadClassBytes(String name) {
        String path = classPath + "/" + name.replace('.', '/') + ".class";
        try {
            java.nio.file.Path p = java.nio.file.Paths.get(path);
            if (!java.nio.file.Files.exists(p)) return null;
            return java.nio.file.Files.readAllBytes(p);
        } catch (Exception e) {
            return null;
        }
    }
}
```

`defineClass` is `ClassLoader`'s key method — it "defines" bytecode into a Class object, filling the method area.

### 4.3 Encrypted Class File Example

```java
public class EncryptedClassLoader extends ClassLoader {
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] encrypted = loadEncryptedBytes(name);
        byte[] decrypted = decrypt(encrypted);   // decrypt
        return defineClass(name, decrypted, 0, decrypted.length);
    }

    private byte[] decrypt(byte[] data) {
        // Simple XOR example (use AES in production)
        for (int i = 0; i < data.length; i++) {
            data[i] ^= 0x42;
        }
        return data;
    }
    // ...
}
```

Encrypt class files first; the custom loader decrypts — protects code from decompilation.

## 5. Class Unloading

A class is unloaded when three conditions are **all met**:

1. **No instances of the class on the heap** — all objects already GC'd.
2. **The class's Class object isn't referenced anywhere** — nobody holds a `Class＜?＞`.
3. **The class's ClassLoader has been reclaimed** — the loader that loaded it is gone.

```java
ClassLoader loader = new MyClassLoader(...);
Class<?> clazz = loader.loadClass("com.example.Foo");
Object obj = clazz.newInstance();

// To unload Foo:
obj = null;       // 1. no instances
clazz = null;     // 2. Class object has no references
loader = null;    // 3. ClassLoader has no references
// After GC, Foo class is unloaded, metaspace memory released
```

The JVM's built-in BootstrapClassLoader never unloads — so JDK-bundled classes (`java.lang.*`, etc.) never unload. This is why metaspace may OOM from "too many dynamically generated classes" — dynamic class unloading is delayed.

## 7. Chapter Summary

| Concept | Key Points |
| --- | --- |
| Loading | Find bytecode, read into memory, generate Class object |
| Verification | Bytecode legality, safety |
| Preparation | Allocate memory for static vars, set zero values |
| Resolution | Symbolic references → direct references |
| Initialization | Execute `＜clinit＞`, static var assignments/static blocks |
| Parent delegation | Child delegates to parent first; parent fails then self loads |
| Bootstrap | Loads JDK core (C++, no parent) |
| Extension/Platform | Loads extensions/JDK modules |
| Application | Loads classpath |
| Breaking parent delegation | Tomcat isolation, SPI reverse loading, hot deployment |
| Custom loader | Override `findClass`, call `defineClass` |
| Class uniqueness | Class name + ClassLoader together identify |
| Unloading conditions | No instances + no Class references + loader reclaimed |

**Mnemonics**:

- **Loading five phases** — loading→verification→preparation→resolution→initialization.
- **Preparation is zero, initialization is assignment** — `static int x = 1` has x=0 in preparation.
- **Parent delegation keeps it safe** — child asks parent first; if parent can't, child loads; prevents faking core classes.
- **Bootstrap is null** — C++ implementation, loads `java.lang.*`.
- **Tomcat breaks parent delegation** — each WebApp has its own loader; app classes first.
- **SPI reverse loading** — `Thread.currentThread().getContextClassLoader()`.
- **Class name + loader = unique identifier** — same bytecode + different loaders = different Class.
- **Unloading needs three absences** — no instances, no Class references, no loader.

## Conclusion: Class Loading Is Java's "Customs"

Class loading is Java's "customs" — every `.class` file must pass through here to enter the JVM and run. Understanding class loading explains:

- Why Tomcat can deploy multiple web apps without mutual interference.
- Why JDBC drivers work just by being on the classpath, no `Class.forName` needed.
- Why hot deployment can "modify without restart."
- Why `ClassNotFoundException` and `NoClassDefFoundError` differ (the former is "not found at load time," the latter "not found at initialization").
