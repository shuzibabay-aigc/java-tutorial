# Security and Authentication

Web application security is serious business — can an unauthenticated user see your orders? Can an ordinary employee access admin endpoints? These are guarded by **authentication** and **authorization** mechanisms. This chapter covers the two mainstream Java security approaches — the Spring Security framework and JWT tokens — plus the OAuth 2.0 protocol.

## 1. Authentication vs Authorization

First, clarify two concepts:

- **Authentication (AuthN)** — Who are you? Verify identity (username/password, SMS, fingerprint). Answers "you are Zhang San."
- **Authorization (AuthZ)** — What can you do? Verify permissions (roles, ACL). Answers "Zhang San is an admin."

The classic trio:

- **Authenticate** — On login, verify the account and password.
- **Session** — After login, issue a credential (Session/JWT); subsequent requests carry it.
- **Authorize** — Once the credential is valid, check whether the target resource can be accessed.

## 2. Spring Security

**Spring Security** is the Spring family's security framework — authentication, authorization, and attack protection (CSRF, CORS, session fixation) all in one. Powerful but with a steep learning curve, because it has many abstraction layers.

### 2.1 The Filter Chain

The core of Spring Security is **FilterChainProxy** — a chain of Servlet Filters, each doing one job:

```
HTTP request
  ↓
SecurityContextPersistenceFilter    -- load/save SecurityContext
HeaderWriterFilter                   -- write security response headers
CorsFilter                           -- CORS handling
LogoutFilter                         -- handle /logout
UsernamePasswordAuthenticationFilter -- handle /login (username/password auth)
JwtAuthenticationFilter              -- (custom) JWT parsing & auth
ExceptionTranslationFilter           -- translate exceptions to 401/403
FilterSecurityInterceptor            -- authorization check
  ↓
Controller
```

Every request passes through this stack of Filters — authentication and authorization both happen in the Filter layer, so by the time the request reaches the Controller everything is already verified.

### 2.2 Configuration

Spring Boot integrates Spring Security:

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>
```

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity   // enable method-level @PreAuthorize
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())           // disable CSRF (not needed for REST APIs)
            .sessionManagement(s -> s.sessionCreationPolicy(
                SessionCreationPolicy.STATELESS))    // stateless (using JWT)
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/login", "/api/register").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .requestMatchers("/api/user/**").authenticated()
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtAuthFilter(),
                UsernamePasswordAuthenticationFilter.class)
            .build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();   // password hashing
    }
}
```

`authorizeHttpRequests` configures path permissions:

- `permitAll()` — accessible to everyone.
- `authenticated()` — accessible after login.
- `hasRole("ADMIN")` — accessible with the ADMIN role.
- `hasAnyRole("ADMIN", "USER")` — any of the roles.
- `hasAuthority("user:write")` — has a specific authority.

### 2.3 Method-level Authorization

```java
@RestController
public class AdminController {

    @PreAuthorize("hasRole('ADMIN')")
    @DeleteMapping("/api/users/{id}")
    public void deleteUser(@PathVariable Long id) { ... }

    @PreAuthorize("#user.id == authentication.principal.id")
    @PutMapping("/api/users/{id}")
    public User update(@PathVariable Long id, @RequestBody User user) { ... }
    // can only update your own info

    @PostAuthorize("returnObject.owner == authentication.name")
    @GetMapping("/api/orders/{id}")
    public Order getOrder(@PathVariable Long id) { ... }
    // check after return: can only view your own orders
}
```

`@PreAuthorize` checks before the method executes; `@PostAuthorize` checks after. SpEL expressions support context variables like `#paramName`, `returnObject`, and `authentication.principal`.

### 2.4 Password Hashing

```java
PasswordEncoder encoder = new BCryptPasswordEncoder();
String raw = "123456";
String hash = encoder.encode(raw);    // $2a$10$xxxxx...
encoder.matches(raw, hash);            // true, verify the password
```

**Never store passwords in plaintext!** `BCrypt` is the industry standard for password hashing — salted, slow hashing (resists rainbow tables and brute force). `MD5`/`SHA1` are insecure — fast hashes are easily brute-forced.

## 3. JWT

**JWT** (JSON Web Token) is a stateless authentication token — the server stores no Session; the token itself carries the user information.

### 3.1 JWT Structure

A JWT consists of three parts, separated by `.`:

```
Header.Payload.Signature
```

- **Header** — Token type and signing algorithm, e.g. `{"alg":"HS256","typ":"JWT"}`.
- **Payload** — User information and claims, e.g. `{"sub":"user42","role":"admin","exp":1700000000}`.
- **Signature** — The signature, `HMACSHA256(base64(header) + "." + base64(payload), secret)`.

```
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyNDIiLCJyb2xlIjoiYWRtaW4ifQ.abc123signature
```

The first two segments are Base64-encoded (not encrypted — they can be decoded); the third is the signature, which prevents tampering.

### 3.2 Workflow

```
1. User logs in (username/password)
       ↓
2. Server verifies -> generates a JWT (containing user info + signature)
       ↓
3. Returns the JWT to the client
       ↓
4. Client stores the JWT (localStorage / cookie)
       ↓
5. Subsequent requests carry Authorization: Bearer <JWT>
       ↓
6. Server verifies the signature -> extracts user info -> handles the request
       ↓
7. No Session needed (stateless)
```

### 3.3 Generating and Verifying JWT

```java
import io.jsonwebtoken.*;
import io.jsonwebtoken.security.Keys;
import java.security.Key;
import java.util.Date;

public class JwtUtil {
    private static final Key KEY = Keys.secretKeyFor(SignatureAlgorithm.HS256);
    private static final long EXPIRATION = 3600_000L;   // 1 hour

    // generate
    public static String generate(Long userId, String role) {
        return Jwts.builder()
            .setSubject(userId.toString())
            .claim("role", role)
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + EXPIRATION))
            .signWith(KEY)
            .compact();
    }

    // verify + parse
    public static Claims parse(String token) {
        return Jwts.parserBuilder()
            .setSigningKey(KEY)
            .build()
            .parseClaimsJws(token)
            .getBody();
    }
}

// usage
String token = JwtUtil.generate(42L, "admin");
Claims claims = JwtUtil.parse(token);
claims.getSubject();    // "42"
claims.get("role");     // "admin"
```

### 3.4 JWT vs Session

| Comparison | Session | JWT |
| --- | --- | --- |
| State | Stored on the server (memory/Redis) | Stateless; the token carries it |
| Scalability | Requires shared Session (Redis) | Naturally distributed-friendly |
| Revocation | Just delete the Session | Hard (needs a blacklist) |
| Size | JSESSIONID Cookie is small | JWT is larger (carries Payload) |
| Security | Server-controlled | Signature prevents tampering, but Payload is readable |

The price of JWT — **hard to revoke proactively**. Once a token is issued, it remains valid until expiry — a password change or logout cannot invalidate it immediately. The workaround is maintaining a "blacklist" (but that breaks statelessness).

### 3.5 JWT Caveats

1. **Do not store sensitive information in the Payload** — Base64 can be decoded; only put non-sensitive data like user_id and role.
2. **Set a short expiry** — around 1 hour, paired with a refresh token for renewal.
3. **Transmit over HTTPS** — if a JWT is intercepted it can be used to impersonate; HTTPS is mandatory.
4. **Keep the signing key secret** — if the key leaks, anyone can forge tokens.
5. **Do not store large amounts of data** — JWT travels with every request; large tokens waste bandwidth.

## 4. OAuth 2.0

**OAuth 2.0** is an authorization framework — letting a user authorize a third-party application to access their resources on another service **without handing over the password**. Think "Log in with GitHub," "Log in with WeChat."

### 4.1 The Four Roles

| Role | Example |
| --- | --- |
| **Resource Owner** | The user |
| **Client** | The third-party application |
| **Authorization Server** | GitHub/WeChat |
| **Resource Server** | GitHub API |

### 4.2 Authorization Code Flow (Most Common)

```
        User        Client        GitHub Auth Server        GitHub Resource Server
         |            |                |                       |
         |---login--->|                |                       |
         |            |---redirect to GitHub login page---->  |
         |            |                |                       |
         |---enter password on GitHub->|                      |
         |            |<--callback with code-|                |
         |            |                |                       |
         |            |--exchange code for token (backend, with secret)---->|
         |            |<--access token-|                       |
         |            |                |                       |
         |            |--call API with token------------------>|
         |            |<--user data----|                       |
```

Core steps:

1. The client redirects the user to the GitHub authorization page (with `client_id` and `redirect_uri`).
2. The user logs in on GitHub and consents to the authorization.
3. GitHub redirects back to the client's `redirect_uri` with a `code` (a one-time short-lived code).
4. The client's **backend** exchanges `code` + `client_secret` for an `access_token` (this happens server-side, so the secret is never exposed).
5. The client uses `access_token` to call the GitHub API and fetch user information.

Why the `code` step? — If the front end received the token directly, `client_secret` would have to be exposed in the front end, which is insecure. `code` is one-time and short-lived; the front end passes `code` to the backend, and the backend exchanges `code` + `secret` for the token — the secret never leaves the backend.

### 4.3 Spring Security OAuth2 Client

```java
@Configuration
public class OAuth2Config {
    @Bean
    public SecurityFilterChain filter(HttpSecurity http) throws Exception {
        return http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/", "/login**").permitAll()
                .anyRequest().authenticated()
            )
            .oauth2Login(Customizer.withDefaults())   // enable OAuth2 login
            .build();
    }
}
```

```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          github:
            client-id: your-github-client-id
            client-secret: your-github-client-secret
            scope: read:user, user:email
          google:
            client-id: xxx
            client-secret: yyy
            scope: profile, email
```

Visiting a page that requires login automatically redirects to the GitHub/Google authorization page — after authorization, the callback automatically establishes a local Session/JWT. This is the principle behind "Log in with GitHub."

    public static void setCurrentUser(Map<String, Object> claims) {
        current.set(claims);

## 6. Other Security Essentials

### 6.1 CSRF

CSRF (Cross-Site Request Forgery) — a malicious site abuses the user's Cookie to impersonate the user and send requests. Spring Security enables CSRF Token protection by default. REST APIs using JWT (no Cookie) are naturally CSRF-resistant and can disable it.

### 6.2 XSS

XSS (Cross-Site Scripting) — a malicious script is injected into a page and executed. Defenses:

- Output escaping (Thymeleaf escapes by default).
- Set `HttpOnly` on cookies (JS cannot read them).
- Content Security Policy (CSP) headers.

### 6.3 CORS

CORS (Cross-Origin Resource Sharing) — the browser's same-origin policy restricts cross-origin requests. Front-end/back-end separation requires configuration:

```java
@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(List.of("https://frontend.com"));
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE"));
    config.setAllowedHeaders(List.of("*"));
    config.setAllowCredentials(true);
    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", config);
    return source;
}
```

### 6.4 HTTPS

Production requires HTTPS — encrypted transport, defense against man-in-the-middle. Let's Encrypt offers free certificates; just configure SSL in Spring Boot.

- **BCrypt for passwords** — slow hashing resists brute force.
- **401 not authenticated, 403 not authorized.**
- **OAuth2 authorization code flow** — code + secret for token.
