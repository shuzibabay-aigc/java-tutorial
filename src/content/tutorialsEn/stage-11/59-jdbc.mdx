# JDBC In Depth

Welcome to Stage 11 — **Database and Networking**. In the previous ten stages our programs stayed inside the JVM — reading and writing files, operating on memory. But real-world applications must talk to databases and communicate over the network. This stage opens two doors: databases (JDBC, connection pools) and networking (Socket, HttpClient). The first door is JDBC — Java's "native tongue" for accessing databases.

JDBC (Java Database Connectivity) is Java's standard API for accessing relational databases. Whether it's MySQL, PostgreSQL, Oracle, or SQLite — as long as the vendor provides a JDBC driver, Java code can operate on it with the same API. This is the "one set of code, many databases" abstraction.

## 1. JDBC Architecture and Driver Types

JDBC is a two-layer architecture:

- **JDBC API** (`java.sql` package) — applications program against it, vendor-agnostic.
- **JDBC Driver API** — vendors implement it, translating JDBC calls into a specific database's protocol.

Drivers come in 4 types (just for understanding; modern usage is all Type 4):

| Type | Principle | Status |
| --- | --- | --- |
| **Type 1** | JDBC-ODBC bridge | Obsolete (removed in Java 8) |
| **Type 2** | Partly Java + native library | Rare |
| **Type 3** | Pure Java, via network middleware | Rare |
| **Type 4** | Pure Java, direct database connection | **Mainstream** (MySQL/PG drivers are this) |

Type 4 is a pure Java implementation, cross-platform, no native library needed — so almost all modern JDBC drivers are Type 4.

## 2. Connecting to a Database

`DriverManager.getConnection` is the entry point for connections:

```java
// Load the driver (auto-loaded since JDBC 4.0+, can be omitted)
Class.forName("com.mysql.cj.jdbc.Driver");

// three arguments: url, username, password
String url = "jdbc:mysql://localhost:3306/test?useSSL=false&serverTimezone=UTC";
try (Connection conn = DriverManager.getConnection(url, "root", "123456")) {
    System.out.println("连接成功: " + conn);
} catch (SQLException e) {
    e.printStackTrace();
}
```

URL format: `jdbc:＜subprotocol＞://＜host＞:＜port＞/＜database＞?＜params＞`. Different databases have different subprotocols — MySQL is `mysql`, PostgreSQL is `postgresql`, Oracle is `oracle:thin`, SQLite is `sqlite`.

Since JDBC 4.0 (Java 6), drivers are auto-discovered via `META-INF/services/java.sql.Driver`, so `Class.forName` can be omitted — as long as the driver jar is on the classpath.

## 3. Statement vs PreparedStatement

There are two main ways to execute SQL — `Statement` and `PreparedStatement`.

### 3.1 Statement: Concatenating SQL Directly (SQL Injection Risk)

```java
Statement stmt = conn.createStatement();
String name = "张三";   // suppose this comes from user input
ResultSet rs = stmt.executeQuery("SELECT * FROM users WHERE name = '" + name + "'");
```

The problem: if the user inputs `' OR '1'='1`, the SQL becomes `SELECT * FROM users WHERE name = '' OR '1'='1` — **SQL injection**! Every user gets returned.

### 3.2 PreparedStatement: Precompiled + Parameterized (Injection-Proof)

```java
String sql = "SELECT * FROM users WHERE name = ?";
PreparedStatement ps = conn.prepareStatement(sql);
ps.setString(1, "张三");    // index starts at 1!
ResultSet rs = ps.executeQuery();
```

`?` is a placeholder, filled via `setString`/`setInt`/`setObject` and friends. The driver escapes the parameters — single quotes become `\'`, so injection fails. **Always use `PreparedStatement`; never concatenate user input with `Statement`!**

```java
ps.setString(1, "' OR '1'='1");   // treated as a plain string, no injection
```

`PreparedStatement` has two extra benefits:

1. **Precompilation** — when the same SQL runs many times, the database compiles the execution plan once, with high performance.
2. **Type safety** — `setInt` on a non-number is a compile error, avoiding type confusion.

## 4. ResultSet and Cursors

`ResultSet` is the query result set — it's a "cursor," initially positioned before the first row; `next()` moves to the next row and returns whether there's data.

```java
ResultSet rs = ps.executeQuery();
while (rs.next()) {           // move to the next row
    int id = rs.getInt("id");          // get by column name
    String name = rs.getString(2);     // get by column index (starts at 1!)
    Date created = rs.getDate("created_at");
    System.out.println(id + ", " + name + ", " + created);
}
```

Note that JDBC indexes **start at 1** (not 0) — a historical relic; many JDBC APIs are 1-based.

By default, a `ResultSet` can only scroll forward with `next()` and is read-only. To make it scrollable and updatable:

```java
Statement stmt = conn.createStatement(
    ResultSet.TYPE_SCROLL_INSENSITIVE,   // scrollable
    ResultSet.CONCUR_UPDATABLE           // updatable
);
// now rs.previous()/rs.first()/rs.absolute(3) etc. all work
```

In practice, scrollable result sets are rarely used — they consume memory for large datasets; usually `LIMIT`/`OFFSET` paging is used instead.

## 5. Batch Processing

To execute large numbers of INSERT/UPDATE, sending them one by one is too slow — `addBatch` + `executeBatch` sends them all at once:

```java
conn.setAutoCommit(false);
PreparedStatement ps = conn.prepareStatement("INSERT INTO logs(msg) VALUES(?)");
for (int i = 0; i < 1000; i++) {
    ps.setString(1, "log " + i);
    ps.addBatch();          // add to batch
    if (i % 100 == 0) {
        ps.executeBatch();   // execute every 100
        ps.clearBatch();
    }
}
ps.executeBatch();           // the remainder
conn.commit();
```

Batch processing compresses 1000 network round trips into 10, a 10x+ performance boost.

## 6. Getting Auto-Increment Primary Keys

After an INSERT, you often need the auto-increment primary key generated by the database:

```java
String sql = "INSERT INTO users(name) VALUES(?)";
PreparedStatement ps = conn.prepareStatement(
    sql,
    Statement.RETURN_GENERATED_KEYS     // declare you want the keys back
);
ps.setString(1, "张三");
ps.executeUpdate();

ResultSet keys = ps.getGeneratedKeys();
if (keys.next()) {
    long id = keys.getLong(1);   // auto-increment primary key
    System.out.println("新用户 id: " + id);
}
```

## 7. Transaction Management

JDBC **auto-commits** by default — every SQL statement commits immediately upon execution. For transaction control, first turn off auto-commit:

```java
Connection conn = DriverManager.getConnection(url, user, pwd);
conn.setAutoCommit(false);   // turn off auto-commit

try {
    // transfer: A loses 100, B gains 100
    ps1.executeUpdate("UPDATE account SET balance = balance - 100 WHERE id = 1");
    ps2.executeUpdate("UPDATE account SET balance = balance + 100 WHERE id = 2");
    conn.commit();            // both succeed -> commit
} catch (SQLException e) {
    conn.rollback();          // any failure -> rollback
    throw e;
}
```

### 7.1 Savepoint: Partial Rollback

Sometimes within a transaction you only want to roll back part of it, leaving earlier successful operations intact — `Savepoint` comes to the rescue:

```java
conn.setAutoCommit(false);
try {
    ps1.executeUpdate("INSERT ...");   // operation 1

    Savepoint sp = conn.setSavepoint("before_risky");   // set a savepoint
    try {
        ps2.executeUpdate("INSERT ...");   // operation 2 (might fail)
    } catch (SQLException e) {
        conn.rollback(sp);    // only roll back to sp, operation 1 preserved
    }

    conn.commit();            // operation 1 + operation 2 (if successful) commit together
} catch (SQLException e) {
    conn.rollback();
}
```

Below is a complete real-JDBC code template (runnable locally with MySQL):

```java
import java.sql.*;

public class JdbcDemo {
    public static void main(String[] args) throws SQLException {
        String url = "jdbc:mysql://localhost:3306/test?useSSL=false&serverTimezone=UTC";
        try (Connection conn = DriverManager.getConnection(url, "root", "123456")) {
            // create table
            try (Statement st = conn.createStatement()) {
                st.execute("CREATE TABLE IF NOT EXISTS users ("
                    + "id INT AUTO_INCREMENT PRIMARY KEY, name VARCHAR(50), age INT)");
            }

            // insert (PreparedStatement + get auto-increment key)
            try (PreparedStatement ps = conn.prepareStatement(
                    "INSERT INTO users(name, age) VALUES(?, ?)",
                    Statement.RETURN_GENERATED_KEYS)) {
                ps.setString(1, "张三");
                ps.setInt(2, 20);
                ps.executeUpdate();
                try (ResultSet keys = ps.getGeneratedKeys()) {
                    if (keys.next()) System.out.println("新 id: " + keys.getLong(1));
                }
            }

            // query
            try (PreparedStatement ps = conn.prepareStatement(
                    "SELECT * FROM users WHERE age > ?")) {
                ps.setInt(1, 18);
                try (ResultSet rs = ps.executeQuery()) {
                    while (rs.next()) {
                        System.out.println(rs.getInt("id") + " "
                            + rs.getString("name") + " " + rs.getInt("age"));
                    }
                }
            }

            // transaction
            conn.setAutoCommit(false);
            try (PreparedStatement ps = conn.prepareStatement(
                    "UPDATE users SET age = age + ? WHERE id = ?")) {
                ps.setInt(1, 1); ps.setInt(2, 1); ps.executeUpdate();
                ps.setInt(1, -1); ps.setInt(2, 2); ps.executeUpdate();
                conn.commit();
            } catch (SQLException e) {
                conn.rollback();
                throw e;
            }
        }
    }
}
```

> **Key observations**:
> - **`PreparedStatement` uses `?` placeholders + `setXxx` to fill parameters** — the core of SQL-injection defense.
> - **JDBC indexes start at 1** — `setString(1, ...)` not `0`; a historical relic.
> - **Transactions in three steps** — `setAutoCommit(false)` → business logic → `commit`/`rollback`.
> - **`try-with-resources` auto-closes** — Connection/Statement/ResultSet must all be closed, otherwise they leak.
> - **Savepoint rolls back to a specific point** — operations before the savepoint are unaffected.

## 9. JDBC Resource Management and Pitfalls

JDBC resources (Connection/Statement/ResultSet) are "heavy" objects that must be closed promptly, or they leak. Best practice:

```java
// recommended: try-with-resources auto-closes
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {
    try (ResultSet rs = ps.executeQuery()) {
        while (rs.next()) { ... }
    }
}
```

**Common pitfalls**:

1. **Not closing a Connection** — the database has a limited number of connections (MySQL default 151); too many leaks cause denial of service.
2. **Not closing a ResultSet** — cursors hold database resources; leaving them open long-term can blow things up.
3. **Creating a Statement inside a loop** — new one each iteration, resources explode.
4. **`SELECT *`** — fetching columns you don't need wastes bandwidth; list columns explicitly.
5. **Transactions crossing methods** — after `setAutoCommit(false)`, calls across methods make it easy to forget commit.

## 10. Chapter Summary

| Concept | Key Points |
| --- | --- |
| `DriverManager.getConnection` | Obtain a connection |
| `Statement` vs `PreparedStatement` | The latter prevents SQL injection |
| `?` placeholder | `setString`/`setInt` fill params, index from 1 |
| `ResultSet` | Cursor-style traversal, `next()` moves |
| Batch processing | `addBatch`/`executeBatch` |
| Auto-increment key | `RETURN_GENERATED_KEYS` + `getGeneratedKeys` |
| Transactions | `setAutoCommit(false)` + `commit`/`rollback` |
| Savepoint | Partial rollback |
| Resource management | `try-with-resources` auto-closes |

**Mnemonic**:

- **Always use `PreparedStatement`** — don't concatenate user input with `Statement`.
- **JDBC indexes start at 1** — historical relic, don't count from 0.
- **Transaction trio** — turn off auto-commit, commit, rollback.
- **Resources use try-with-resources** — Connection/Statement/ResultSet all need closing.
- **Batch: add then execute** — 10x performance for bulk inserts.
