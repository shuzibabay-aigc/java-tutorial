# HttpClient

In the previous chapter we wrote TCP communication with raw Sockets — able to send byte streams, but you have to assemble HTTP messages yourself, parse status codes, handle chunked encoding… exhausting. HTTP is the internet's most-used application-layer protocol, and Java naturally has a ready-made API for it. This chapter looks at Java 11+'s modern **HttpClient** — synchronous and asynchronous in one, with an elegant API and solid performance.

## 1. The Problems with the Old HttpURLConnection

Java's early built-in HTTP client was `HttpURLConnection` (Java 1.1, 1997):

```java
URL url = new URL("https://api.example.com/users");
HttpURLConnection conn = (HttpURLConnection) url.openConnection();
conn.setRequestMethod("GET");
conn.setRequestProperty("Accept", "application/json");
int code = conn.getResponseCode();
try (BufferedReader r = new BufferedReader(
        new InputStreamReader(conn.getInputStream()))) {
    String line;
    while ((line = r.readLine()) != null) System.out.println(line);
}
```

Its ailments:

1. **Clunky API** — `URLConnection` was designed in 1997, verbose naming, unfriendly types.
2. **Blocking by default** — no async support; high concurrency requires your own thread pool.
3. **Poor HTTP/2 support** — only HTTP/1.1; can't use HTTP/2 multiplexing.
4. **Unintuitive API** — `getResponseCode`/`getInputStream` scattered around, no Builder pattern.
5. **Stagnant maintenance** — Oracle barely updates it anymore.

The industry used third-party libraries like Apache HttpClient and OkHttp. The new `java.net.http.HttpClient`, introduced in Java 9 and standardized in Java 11, finally offers an official modern solution.

## 2. HttpClient API Overview

Three core classes of the new API:

| Class | Purpose |
| --- | --- |
| `HttpClient` | The client; manages the connection pool, HTTP version, timeouts, etc. |
| `HttpRequest` | The request; carries URL, method, headers, body |
| `HttpResponse＜T＞` | The response; carries status code, headers, body (T is the body type) |

The design is **Builder pattern + immutable** — both HttpClient and HttpRequest are immutable objects, thread-safe after construction, and reusable.

```java
import java.net.URI;
import java.net.http.*;
import java.time.Duration;

// 1. Create an HttpClient (reusable, thread-safe)
HttpClient client = HttpClient.newBuilder()
    .version(HttpClient.Version.HTTP_2)        // prefer HTTP/2
    .connectTimeout(Duration.ofSeconds(10))    // connection timeout
    .followRedirects(HttpClient.Redirect.NORMAL)  // follow redirects
    .build();

// 2. Build the request
HttpRequest req = HttpRequest.newBuilder()
    .uri(URI.create("https://api.github.com/users/octocat"))
    .timeout(Duration.ofSeconds(10))
    .header("Accept", "application/json")
    .header("User-Agent", "Java HttpClient")
    .GET()    // can be omitted, GET is the default
    .build();

// 3. Send the request (synchronous)
HttpResponse<String> resp = client.send(req, HttpResponse.BodyHandlers.ofString());

System.out.println(resp.statusCode());         // 200
System.out.println(resp.headers().firstValue("Content-Type"));
System.out.println(resp.body());               // JSON string
```

`HttpClient` is a heavy object (it holds a connection pool internally); **build one globally and reuse it** — don't new one per request.

## 3. Synchronous Request: send

`send` blocks the current thread until the response comes back:

```java
HttpResponse<String> resp = client.send(req, HttpResponse.BodyHandlers.ofString());
```

The second argument is a `BodyHandler＜T＞` — telling it how to handle the response body:

| BodyHandler | Handling |
| --- | --- |
| `ofString()` | As a string |
| `ofByteArray()` | As a byte array |
| `ofFile(Path)` | Write directly to a file (good for large downloads) |
| `ofInputStream()` | As an InputStream (streaming) |
| `ofLines()` | As a `Stream＜String＞` (line by line) |
| `discarding()` | Discard the body (only care about the status code) |

For large file downloads, use `ofFile` to avoid memory blowup:

```java
HttpRequest req = HttpRequest.newBuilder()
    .uri(URI.create("https://example.com/bigfile.zip"))
    .build();
Path file = Path.of("download.zip");
HttpResponse<Path> resp = client.send(req, HttpResponse.BodyHandlers.ofFile(file));
```

## 4. Asynchronous Request: sendAsync

`sendAsync` immediately returns a `CompletableFuture＜HttpResponse＜T＞＞` — non-blocking; the callback fires after the response arrives:

```java
CompletableFuture<HttpResponse<String>> future =
    client.sendAsync(req, HttpResponse.BodyHandlers.ofString());

future.thenAccept(resp -> {
    System.out.println("状态码: " + resp.statusCode());
    System.out.println("Body: " + resp.body());
}).exceptionally(e -> {
    System.err.println("请求失败: " + e.getMessage());
    return null;
});

// the main thread goes on doing other things...
future.join();   // block and wait when needed
```

The power of async lies in **concurrent requests** — fire 10 requests at once, and the total time ≈ the slowest one, not the sum of all 10:

```java
List<URI> uris = List.of(uri1, uri2, uri3, /* ... */);

List<CompletableFuture<String>> futures = uris.stream()
    .map(uri -> client.sendAsync(
            HttpRequest.newBuilder(uri).build(),
            HttpResponse.BodyHandlers.ofString())
        .thenApply(HttpResponse::body))
    .toList();

CompletableFuture<Void> all = CompletableFuture.allOf(
    futures.toArray(new CompletableFuture[0]));
all.join();   // wait for all to complete

List<String> results = futures.stream()
    .map(CompletableFuture::join)
    .toList();
```

10 requests fired concurrently, 10x faster than serial — that's the value of async.

## 5. POST/PUT/DELETE and Body

To send a body, use `HttpRequest.BodyPublishers`:

```java
import static java.net.http.HttpRequest.BodyPublishers.*;

// POST JSON
String json = "{\"name\":\"张三\",\"age\":20}";
HttpRequest postReq = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/users"))
    .header("Content-Type", "application/json")
    .POST(ofString(json))           // string body
    .build();

// POST form
String form = "name=张三&age=20";
HttpRequest formReq = HttpRequest.newBuilder()
    .uri(URI.create("https://example.com/login"))
    .header("Content-Type", "application/x-www-form-urlencoded")
    .POST(ofString(form))
    .build();

// POST a file (upload)
HttpRequest uploadReq = HttpRequest.newBuilder()
    .uri(URI.create("https://example.com/upload"))
    .POST(ofFile(Path.of("data.bin")))
    .build();

// PUT
HttpRequest putReq = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/users/1"))
    .PUT(ofString(json))
    .build();

// DELETE
HttpRequest delReq = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/users/1"))
    .DELETE()
    .build();
```

The `BodyPublishers` factories:

- `ofString(s)` — a string.
- `ofString(s, charset)` — a string with a specified charset.
- `ofByteArray(bytes)` — a byte array.
- `ofFile(path)` — a file (streaming, no memory burden).
- `ofInputStream(supplier)` — read from an InputStream.
- `noBody()` — no body.

## 6. Timeouts, Headers, and Configuration

### 6.1 Timeouts

Two layers of timeout:

```java
HttpClient client = HttpClient.newBuilder()
    .connectTimeout(Duration.ofSeconds(5))   // connection-establishment timeout
    .build();

HttpRequest req = HttpRequest.newBuilder()
    .uri(uri)
    .timeout(Duration.ofSeconds(10))         // whole-request timeout
    .build();
```

`connectTimeout` limits the TCP handshake; `timeout` limits the whole request (including reading the body). On timeout, `HttpTimeoutException` is thrown.

### 6.2 Headers

```java
HttpRequest req = HttpRequest.newBuilder()
    .uri(uri)
    .header("Accept", "application/json")
    .header("Authorization", "Bearer " + token)
    .header("Content-Type", "application/json; charset=utf-8")
    .build();

// multiple values with the same name use headers
.headers("X-Custom", "v1", "X-Custom", "v2")

// or setHeader to overwrite
.setHeader("User-Agent", "MyApp/1.0")
```

Reading response headers:

```java
HttpResponse<String> resp = client.send(req, ofString());
Optional<String> ct = resp.headers().firstValue("Content-Type");
List<String> cookies = resp.headers().allValues("Set-Cookie");
```

### 6.3 HTTP/2 and Redirects

```java
HttpClient client = HttpClient.newBuilder()
    .version(HttpClient.Version.HTTP_2)             // prefer HTTP/2 (default)
    .followRedirects(HttpClient.Redirect.NORMAL)    // follow redirects (same protocol)
    .proxy(ProxySelector.of(new InetSocketAddress("proxy.host", 8080)))  // proxy
    .authenticator(new Authenticator() {            // authentication
        @Override protected PasswordAuthentication getPasswordAuthentication() {
            return new PasswordAuthentication("user", "pwd".toCharArray());
        }
    })
    .build();
```

`Redirect` has three levels: `NEVER` (don't follow), `NORMAL` (follow same protocol), `ALWAYS` (follow even across protocols, e.g., HTTPS→HTTP, which has security risks).

## 7. WebSocket

Java 9+'s `java.net.http.WebSocket` provides a native WebSocket client:

```java
import java.net.http.WebSocket;
import java.net.URI;

WebSocket ws = HttpClient.newHttpClient()
    .newWebSocketBuilder()
    .connectTimeout(Duration.ofSeconds(5))
    .buildAsync(URI.create("wss://echo.example.com/socket"),
        new WebSocket.Listener() {
            @Override public void onOpen(WebSocket webSocket) {
                System.out.println("[WS] 连接打开");
                webSocket.sendText("Hello WebSocket", true);
            }
            @Override public CompletionStage<?> onText(WebSocket webSocket,
                    CharSequence data, boolean last) {
                System.out.println("[WS] 收到: " + data);
                return null;
            }
            @Override public void onError(WebSocket webSocket, Throwable error) {
                System.err.println("[WS] 错误: " + error);
            }
            @Override public CompletionStage<?> onClose(WebSocket webSocket,
                    int statusCode, String reason) {
                System.out.println("[WS] 关闭: " + statusCode + " " + reason);
                return null;
            }
        })
    .join();

ws.sendText("ping", true);
ws.sendClose(WebSocket.NORMAL_CLOSURE, "bye");
```

WebSocket is bidirectional, full-duplex communication — the server can proactively push messages, making it suitable for chat, real-time notifications, and stock quotes. `Listener` is a callback interface; each event maps to a method.

> **Key observations**:
> - **HttpClient reuses a single instance** — it holds a connection pool internally; don't new one each time.
> - **Synchronous `send`, asynchronous `sendAsync`** — async returns a `CompletableFuture`, chainable with `thenAccept`/`exceptionally`.
> - **5 concurrent requests take ≈ the time of a single request** — async concurrency is HttpClient's killer feature.
> - **Status codes 201/404 are both accessible** — use `statusCode()` to determine the business outcome.
> - **Exceptions go to `exceptionally`** — timeouts, connection failures, and other exceptions don't enter `thenAccept`; handle them separately.
> - **`BodyHandlers.ofString` treats the body as a string** — for large files, use `ofFile` to stream to disk.

## 9. HttpClient vs Third-Party Libraries

| Library | Pros | Cons |
| --- | --- | --- |
| **Java HttpClient** | Built-in, no dependency, modern API, HTTP/2 | Relatively simple feature set |
| **Apache HttpClient** | Feature-rich, mature ecosystem | Slightly verbose API, heavy dependency |
| **OkHttp** | Elegant API, good performance, mainstream on Android | Requires a dependency |
| **Retrofit** | Declarative REST client (built on OkHttp) | Requires a dependency |

**Selection advice**:

- Simple project, don't want a dependency → Java HttpClient.
- Spring Boot project → use `RestTemplate`/`WebClient` (backed by HttpClient/Jetty underneath).
- Complex needs (interceptors, complex retries) → Apache HttpClient or OkHttp.
- Declarative REST (Feign-style) → Spring Cloud OpenFeign.

## 10. Chapter Summary

| Concept | Key Points |
| --- | --- |
| `HttpClient` | The client, reusable, holds a connection pool |
| `HttpRequest` | The request, built with a Builder, immutable |
| `HttpResponse＜T＞` | The response; T is determined by the BodyHandler |
| `send` | Synchronous, blocking |
| `sendAsync` | Asynchronous, returns `CompletableFuture` |
| `BodyHandlers` | Handle the response body (ofString/ofFile/...) |
| `BodyPublishers` | Produce the request body (ofString/ofFile/...) |
| `WebSocket` | Bidirectional full-duplex communication, Listener callbacks |

**Mnemonic**:

- **HttpClient reused globally** — don't new one each time; the connection pool would go to waste.
- **Synchronous send, asynchronous sendAsync** — async returns a `CompletableFuture`.
- **BodyHandler handles the response** — ofString for strings, ofFile to disk.
- **Use allOf for concurrency** — fire N at once, total time ≈ the slowest.
- **Two timeout layers** — connectTimeout for the connection, timeout for the whole request.
