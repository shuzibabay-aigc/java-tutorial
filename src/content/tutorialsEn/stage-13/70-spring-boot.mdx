# Spring Boot

The previous chapter covered Spring's IoC/AOP — powerful but configuration-heavy: you had to write piles of XML or Java Config, deploy to Tomcat, and juggle dependency versions. **Spring Boot** was built to liberate you from all that — "convention over configuration" + "ready out of the box," turning a Spring app from "a week to set up" into "running in five minutes."

This chapter covers Spring Boot's core — auto-configuration, starter dependencies, REST APIs, configuration files, data-access integration, and virtual threads.

## 1. What Spring Boot Solves

Pain points of traditional Spring applications:

1. **Configuration explosion** — A simple web app needed `web.xml`, `applicationContext.xml`, `spring-mvc.xml`, `dispatcher-servlet.xml`...
2. **Dependency hell** — Spring itself has dozens of modules; add Spring MVC, transactions, ORM, JSON... and you must reconcile versions yourself.
3. **Tedious deployment** — Package as a WAR, deploy to Tomcat, configure the data source...
4. **Slow startup** — Tomcat bootstrap + Spring context loading, tens of seconds.

Spring Boot's remedies:

- **Starter dependencies** — A single `spring-boot-starter-web` pulls in every library needed for web development, with versions already reconciled.
- **Auto-configuration** — Add a database driver, and the data source is configured automatically; add Thymeleaf, and the view resolver is wired up.
- **Embedded Tomcat** — `java -jar app.jar` runs directly, no deployment needed.
- **Actuator** — Built-in health checks and monitoring endpoints.
- **Production-ready** — Default settings suit most scenarios out of the box.

## 2. Starter Dependencies and Auto-Configuration

### 2.1 Starter Dependencies

Spring Boot packages the dependencies for common scenarios into "Starters" — one dependency pulls in the whole set:

| Starter | Purpose |
| --- | --- |
| `spring-boot-starter` | Core foundation |
| `spring-boot-starter-web` | Web + embedded Tomcat + Spring MVC |
| `spring-boot-starter-data-jpa` | Spring Data JPA + Hibernate |
| `spring-boot-starter-data-redis` | Redis integration |
| `spring-boot-starter-security` | Spring Security |
| `spring-boot-starter-test` | Testing (JUnit/Mockito/AssertJ) |
| `spring-boot-starter-actuator` | Monitoring endpoints |
| `spring-boot-starter-validation` | Bean Validation |

```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.0</version>
</parent>

<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
</dependencies>
```

`spring-boot-starter-parent` is the parent POM — it manages the versions of all Spring Boot-related dependencies, so child projects need not specify versions. This is "version reconciliation."

### 2.2 How Auto-Configuration Works

`@SpringBootApplication` is a composed annotation:

```java
@SpringBootApplication   // equivalent to the three below
@SpringBootConfiguration   // @Configuration, a configuration class
@EnableAutoConfiguration   // enable auto-configuration
@ComponentScan             // scan the current package and sub-packages
public class App { ... }
```

`@EnableAutoConfiguration` is the heart of auto-configuration — it loads a batch of auto-configuration classes via `spring.factories` (Spring Boot 2.x) or `META-INF/spring/...imports` (3.x). Each configuration class decides whether to take effect based on conditions:

```java
@AutoConfiguration
@ConditionalOnClass(DataSource.class)              // active only if DataSource is on the classpath
@ConditionalOnProperty(prefix = "spring.datasource",
                       name = "url")                // active only if url is configured
@EnableConfigurationProperties(DataSourceProperties.class)
public class DataSourceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean                       // create only if the user has not defined one
    public DataSource dataSource(DataSourceProperties props) {
        return props.initializeDataSourceBuilder().build();
    }
}
```

`@ConditionalOnXxx` is the key — conditional wiring:

- `@ConditionalOnClass(X.class)` — active only if class X is on the classpath.
- `@ConditionalOnMissingBean` — active only if the user has not defined one (user's takes priority).
- `@ConditionalOnProperty` — active only if a given property is configured.
- `@ConditionalOnWebApplication` — active only if this is a web application.

The effect — add the MySQL driver + set `spring.datasource.url`, and the data source is configured; do not add it, and nothing happens. **"Convention over configuration"** — do things the conventional way with zero configuration; only deviate from convention when you must configure.

## 3. Your First REST API

```java
// src/main/java/com/example/App.java
@SpringBootApplication
public class App {
    public static void main(String[] args) {
        SpringApplication.run(App.class, args);
    }
}
```

```java
// src/main/java/com/example/controller/UserController.java
@RestController
@RequestMapping("/api/users")
@RequiredArgsConstructor
public class UserController {

    private final UserService userService;

    @GetMapping
    public List<User> list() {
        return userService.findAll();
    }

    @GetMapping("/{id}")
    public User getById(@PathVariable Long id) {
        return userService.findById(id);
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public User create(@Valid @RequestBody CreateUserRequest req) {
        return userService.create(req);
    }

    @PutMapping("/{id}")
    public User update(@PathVariable Long id, @RequestBody UpdateUserRequest req) {
        return userService.update(id, req);
    }

    @DeleteMapping("/{id}")
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public void delete(@PathVariable Long id) {
        userService.delete(id);
    }
}
```

Annotation breakdown:

| Annotation | Effect |
| --- | --- |
| `@RestController` | `@Controller` + `@ResponseBody`; the return value is auto-serialized to JSON |
| `@RequestMapping` | Class-level path prefix |
| `@GetMapping`/`@PostMapping`/... | HTTP method mapping |
| `@PathVariable` | Extract a URL path variable `/users/{id}` |
| `@RequestParam` | Extract a query parameter `?name=x` |
| `@RequestBody` | Deserialize the request body into an object |
| `@Valid` | Trigger Bean Validation |
| `@ResponseStatus` | Specify the response status code |

After startup, `curl http://localhost:8080/api/users` returns JSON — Spring Boot auto-configured Jackson serialization, the embedded Tomcat, and DispatcherServlet, all with zero configuration.

## 4. The application.yml Configuration File

Spring Boot supports both `application.properties` and `application.yml`. yml's hierarchy is clearer and is the mainstream choice:

```yaml
server:
  port: 8080                    # server port
  servlet:
    context-path: /api          # context path

spring:
  datasource:
    url: jdbc:mysql://localhost:3306/test
    username: root
    password: 123456
    driver-class-name: com.mysql.cj.jdbc.Driver
    hikari:
      maximum-pool-size: 10
      minimum-idle: 2

  jpa:
    hibernate:
      ddl-auto: update          # auto-create tables
    show-sql: true
    properties:
      hibernate.format_sql: true

  redis:
    host: localhost
    port: 6379

  threads:
    virtual:
      enabled: true             # Spring Boot 3.2+ enable virtual threads

logging:
  level:
    com.example: DEBUG
    org.hibernate.SQL: DEBUG
  file:
    name: logs/app.log

management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,env
```

**Profile for multiple environments** — `application-dev.yml`/`application-prod.yml`; activate with `spring.profiles.active=prod`.

Binding to fields:

```java
@ConfigurationProperties(prefix = "app")
@Data
public class AppProperties {
    private String name;
    private int maxSize;
    private List<String> allowedOrigins;
}
```

```yaml
app:
  name: 我的系统
  max-size: 100
  allowed-origins:
    - https://a.com
    - https://b.com
```

## 5. Integrating MyBatis / MyBatis-Plus

```xml
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>mybatis-plus-boot-starter</artifactId>
    <version>3.5.5</version>
</dependency>
```

```java
@Data
@TableName("users")
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String name;
    private Integer age;
}

@Mapper
public interface UserMapper extends BaseMapper<User> {
    // BaseMapper provides: insert/updateById/deleteById/selectById/selectList...
    // Write complex queries in XML or annotations
    @Select("SELECT * FROM users WHERE age > #{age}")
    List<User> findOlderThan(int age);
}

@Service
@RequiredArgsConstructor
public class UserService {
    private final UserMapper userMapper;

    public List<User> findAll() {
        return userMapper.selectList(null);   // null = no condition
    }

    public User findById(Long id) {
        return userMapper.selectById(id);
    }

    public User create(User u) {
        userMapper.insert(u);
        return u;   // u.id is auto-populated
    }

    // Condition builder
    public List<User> findByNameLike(String keyword) {
        return userMapper.selectList(
            new LambdaQueryWrapper<User>()
                .like(User::getName, keyword)
                .ge(User::getAge, 18)
                .orderByDesc(User::getId)
        );
    }
}
```

MyBatis-Plus's killer feature — `BaseMapper` provides CRUD out of the box, and complex queries are built fluently with `LambdaQueryWrapper`. Zero SQL covers most scenarios.

## 6. Integrating Spring Data JPA

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>
```

```java
@Entity
@Table(name = "users")
@Data
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;
    private Integer age;
}

// Repository interface, no implementation needed!
public interface UserRepository extends JpaRepository<User, Long> {
    List<User> findByAgeGreaterThan(int age);
    Optional<User> findByName(String name);

    @Query("SELECT u FROM User u WHERE u.age BETWEEN :min AND :max")
    List<User> findByAgeRange(@Param("min") int min, @Param("max") int max);
}

@Service
@RequiredArgsConstructor
public class UserService {
    private final UserRepository repo;

    @Transactional
    public User create(User u) {
        return repo.save(u);
    }

    public List<User> findByAgeRange(int min, int max) {
        return repo.findByAgeRange(min, max);
    }
}
```

Spring Data JPA's "magic" — just declare interface methods, and the framework implements them. A method name like `findByAgeGreaterThan` is parsed directly into SQL `WHERE age ＞ ?`. Use `@Query` for complex queries with JPQL.

## 7. Spring Boot and Virtual Threads

Spring Boot 3.2+ supports virtual threads (Java 21) — one line of config lets every HTTP request run on a virtual thread:

```yaml
spring:
  threads:
    virtual:
      enabled: true
```

The effect — Tomcat is no longer bound by the platform thread pool (default 200); each request gets a virtual thread, easily handling 100,000 concurrent connections. Virtual threads automatically yield the CPU during IO and consume almost no resources.

Traditional model (platform threads):

```
Request1 → Thread1 (blocks on IO, Thread1 just waits)
Request2 → Thread2
...
Request200 → Thread200 (pool exhausted, subsequent requests queue up)
```

Virtual thread model:

```
Request1 → VirtualThread1 (suspends on IO, the platform thread runs other virtual threads)
Request2 → VirtualThread2
...
Request100000 → VirtualThread100000 (all running on a few dozen platform threads)
```

**Note** — virtual threads are not for CPU-intensive tasks (they solve the IO-blocking problem). Synchronous blocking code (JDBC, HTTP calls) benefits directly, with no code changes.

## 8. Actuator Monitoring

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,env,loggers
  endpoint:
    health:
      show-details: always
```

Common endpoints:

| Endpoint | Purpose |
| --- | --- |
| `/actuator/health` | Health check (database, Redis status) |
| `/actuator/info` | Application info |
| `/actuator/metrics` | Metrics (JVM, HTTP requests) |
| `/actuator/env` | Environment variables |
| `/actuator/loggers` | Dynamically adjust log levels |
| `/actuator/beans` | All Beans |
| `/actuator/mappings` | All URL mappings |

```bash
curl localhost:8080/actuator/health
# {"status":"UP","components":{"db":{"status":"UP"},...}}

# Dynamically adjust log level (no restart needed)
curl -X POST localhost:8080/actuator/loggers/com.example \
  -H 'Content-Type: application/json' \
  -d '{"configuredLevel":"DEBUG"}'
```

In production, pair with Prometheus + Grafana for monitoring — `/actuator/prometheus` exposes Prometheus-format metrics.

## 9. Hands-on: Simulating Spring Boot Auto-Configuration

Because Spring Boot requires a full runtime, Piston cannot run it. Below we use plain Java SE to simulate the core idea of **auto-configuration** — deciding which Beans to create based on what classes are on the classpath and what properties are in the config file.

import java.lang.reflect.*;
import java.util.*;
import java.util.concurrent.*;

// === 模拟 Spring Boot 注解 ===
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
@interface ConditionalOnClass {
    Class<?> value();

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
@interface ConditionalOnProperty {
    String name();

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
@interface ConditionalOnMissingBean {
    Class<?> value();

// === 模拟自动配置类 ===

@AutoConfiguration
@ConditionalOnClass(DataSource.class)
class DataSourceAutoConfiguration {

// === 迷你 Spring Boot 容器 ===
class MiniSpringBoot {
    private final Map<String, Object> beans = new ConcurrentHashMap<>();
    private final Map<String, String> properties = new ConcurrentHashMap<>();
    private final Set<Class<?>> classpathClasses = new HashSet<>();

    public void setProperty(String k, String v) { properties.put(k, v); }
    public void registerClasspath(Class<?> c) { classpathClasses.add(c); }

    @SuppressWarnings("unchecked")
    public <T> T getBean(Class<T> type) {
        return (T) beans.values().stream()
            .filter(v -> type.isInstance(v))
            .findFirst().orElse(null);
    }

    // 执行自动配置: 扫描所有 @AutoConfiguration 类
    public void runAutoConfiguration(Class<?>... configs) throws Exception {
        System.out.println("[Spring Boot] 启动自动配置...");
        System.out.println("  classpath 类: " + classpathClasses);
        System.out.println("  配置: " + properties);

        for (Class<?> configClass : configs) {
            if (!configClass.isAnnotationPresent(AutoConfiguration.class)) continue;

            // 检查 @ConditionalOnClass
            ConditionalOnClass onClass = configClass.getAnnotation(ConditionalOnClass.class);
            if (onClass != null && !classpathClasses.contains(onClass.value())) {
                System.out.println("  [跳过] " + configClass.getSimpleName()
                    + " (classpath 缺少 " + onClass.value().getSimpleName() + ")");
                continue;
            }

            Object configInstance = configClass.getDeclaredConstructor().newInstance();
            System.out.println("  [生效] " + configClass.getSimpleName());

            // 执行 @Bean 方法
            for (Method m : configClass.getDeclaredMethods()) {
                if (!m.isAnnotationPresent(Bean.class)) continue;

                // @ConditionalOnProperty
                ConditionalOnProperty onProp = m.getAnnotation(ConditionalOnProperty.class);
                if (onProp != null && !properties.containsKey(onProp.name())) {
                    System.out.println("    [跳过] " + m.getName()
                        + " (未配 " + onProp.name() + ")");
                    continue;
                }

                // @ConditionalOnMissingBean
                ConditionalOnMissingBean onMissing = m.getAnnotation(ConditionalOnMissingBean.class);
                if (onMissing != null && getBean(onMissing.value()) != null) {
                    System.out.println("    [跳过] " + m.getName() + " (已存在)");
                    continue;
                }

                // 调用方法创建 Bean
                Object bean = m.invoke(configInstance, properties);
                beans.put(m.getName(), bean);
                System.out.println("    [创建] " + m.getName() + " -> " + bean);
            }
        }
    }
}

## 10. Chapter Summary

| Concept | Key Points |
| --- | --- |
| Starter | Starter dependency; one pulls in a whole set of libraries |
| Auto-configuration | Conditional wiring via `@ConditionalOnXxx` |
| `@SpringBootApplication` | Configuration + auto-configuration + component scanning |
| `@RestController` | `@Controller` + `@ResponseBody` |
| `application.yml` | Configuration file; supports Profiles |
| MyBatis-Plus | `BaseMapper` provides CRUD + condition builder |
| Spring Data JPA | Method names parsed into SQL; zero implementation |
| Virtual threads | Enabled with one line in Spring Boot 3.2+ |
| Actuator | Monitoring endpoints; health checks/metrics |

**Memory mnemonics**:

- **Starter pulls in dependencies** — `spring-boot-starter-web` spins up a web app in one line.
- **Auto-configuration wires conditionally** — `@ConditionalOnClass`/`Property`/`MissingBean`.
- **Embedded Tomcat** — `java -jar` runs directly.
- **`@RestController` returns JSON** — Jackson serialization automatically.
- **MyBatis-Plus `BaseMapper`** — CRUD with zero code.
- **JPA method names are SQL** — `findByAgeGreaterThan` is parsed automatically.
- **Virtual threads in one line** — `spring.threads.virtual.enabled=true`.
