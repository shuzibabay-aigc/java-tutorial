# Logging Frameworks

`System.out.println` is every beginner's favorite debugging tool, but using it in production is a disaster — no level filtering, can't write to files, can't roll archives, can't control format, and poor performance (synchronous IO blocking). **Logging frameworks** were born to solve these — they are the "black box recorder" of production-grade applications, letting you reconstruct the scene from a pile of logs when something goes wrong.

This chapter covers Java logging's two-layer "facade + implementation" architecture — SLF4J facade + Logback implementation — and logging best practices.

## 1. The Messy History of Java Logging

The Java logging ecosystem is messy; let's sort it out:

| Framework | Role | Era |
| --- | --- | --- |
| `java.util.logging` (JUL) | Built into the JDK | 1999 |
| **Log4j 1.x** | First-generation mainstream | 2001 |
| **SLF4J** | Logging **facade** (API) | 2004 |
| **Logback** | New implementation by the Log4j author | 2009 |
| **Log4j 2** | Log4j rewrite, best performance | 2014 |

The root of the mess — Log4j 1.x was the early mainstream but didn't design a facade; JUL ships with the JDK but has an awkward API; Apache Commons Logging (JCL) tried to be a facade but had classloader pitfalls; finally SLF4J emerged as the facade standard, with Logback/Log4j2 as implementations.

**Modern best practice**:

```
Application code --> SLF4J (facade) --> Logback or Log4j2 (implementation)
```

The application only programs against the SLF4J API; which implementation sits behind it is decided by configuration. Swapping implementations requires no business code changes — this is the "facade pattern".

## 2. SLF4J: The Logging Facade

**SLF4J** (Simple Logging Facade for Java) is only an API and produces no log output itself. It needs to bind to an underlying implementation:

```xml
<!-- SLF4J API -->
<dependency>
    <groupId>org.slf4j</groupId>
    <artifactId>slf4j-api</artifactId>
    <version>2.0.9</version>
</dependency>

<!-- Bind Logback (Logback ships with its own slf4j binding, no extra dependency needed) -->
<dependency>
    <groupId>ch.qos.logback</groupId>
    <artifactId>logback-classic</artifactId>
    <version>1.4.14</version>
</dependency>
```

### 2.1 Basic Usage

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class UserService {
    // Declare the logger with the class name for easy locating
    private static final Logger log = LoggerFactory.getLogger(UserService.class);

    public void login(String name) {
        log.info("User logged in: {}", name);          // Parameterized logging
        log.debug("Debug info: id={}", 42);

        try {
            // Business logic...
        } catch (Exception e) {
            log.error("Login failed: user={}", name, e);   // Exception as the last argument
        }
    }
}
```

`{}` is a placeholder — arguments fill in by order. This is one of SLF4J's killer features (see the performance section below).

## 3. Log Levels

Logs come in 5 levels, from the most verbose to the most severe:

| Level | Purpose | Example |
| --- | --- | --- |
| `TRACE` | Extremely fine-grained tracing | `Entering method foo with args=[1,2]` |
| `DEBUG` | Debugging information | `Cache miss for key=user:42` |
| `INFO` | Important business events | `User 42 logged in from 1.2.3.4` |
| `WARN` | Warning, recoverable | `Connection pool 80% full` |
| `ERROR` | Error, affects functionality | `Failed to send email to user 42` |

Levels are **hierarchical** — enabling `INFO` outputs `INFO`/`WARN`/`ERROR` but not `TRACE`/`DEBUG`. Production usually enables `INFO`; development enables `DEBUG`.

```java
log.trace("Most verbose");    // Almost never on in production
log.debug("For debugging");   // On in dev, off in prod
log.info("Important event");  // Default level
log.warn("Warning");
log.error("Error");           // Must read
```

**How to choose levels?**

- **ERROR** — affects functionality, needs immediate attention (database unreachable, critical call failed).
- **WARN** — recoverable but noteworthy (retry succeeded, degraded, connection pool nearing limit).
- **INFO** — key business milestones (login, order placed, payment completed).
- **DEBUG** — for troubleshooting during development; turn off in production.
- **TRACE** — rarely used, almost only briefly when debugging a specific bug.

**Common misuses**:

- Logging all exceptions as ERROR — some exceptions are normal business branches (like `UserNotFoundException`), which should be WARN or INFO.
- Too much INFO — log explosion buries key information. Be restrained with INFO.
- Logging ERROR and then throwing — the upper layer logs ERROR again, duplicating. Either log or throw, not both.

## 4. Logback Configuration

`logback.xml` (placed in `src/main/resources/`) controls log format, output destinations, and levels:

```xml
<configuration>
    <!-- Console output -->
    <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- File output (rolling archive) -->
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/app.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
            <fileNamePattern>logs/app.%d{yyyy-MM-dd}.%i.log.gz</fileNamePattern>
            <maxFileSize>100MB</maxFileSize>
            <maxHistory>30</maxHistory>
            <totalSizeCap>10GB</totalSizeCap>
        </rollingPolicy>
        <encoder>
            <pattern>%d{ISO8601} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- Control level by package name -->
    <logger name="com.example.service" level="DEBUG"/>
    <logger name="org.springframework" level="WARN"/>
    <logger name="org.hibernate.SQL" level="DEBUG"/>

    <!-- Root level -->
    <root level="INFO">
        <appender-ref ref="STDOUT"/>
        <appender-ref ref="FILE"/>
    </root>
</configuration>
```

### 4.1 Pattern Format

`%pattern` controls the format of each log line:

| Placeholder | Meaning |
| --- | --- |
| `%d{...}` | Timestamp, format like `SimpleDateFormat` |
| `%thread` | Thread name |
| `%-5level` | Level (left-aligned, 5 chars) |
| `%logger{36}` | Logger name (class name), `{36}` is the abbreviation length |
| `%msg` / `%message` | Log message |
| `%n` | Newline |
| `%M` | Method name (high performance cost, use sparingly) |
| `%L` | Line number (high performance cost, use sparingly) |
| `%X{key}` | MDC (Mapped Diagnostic Context) value |
| `%-20.20logger` | Logger name fixed to 20 chars wide |

Classic Pattern:

```
%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n
```

Output:

```
2024-01-15 14:30:25.123 [main] INFO  c.e.s.UserService - User logged in: Alice
```

### 4.2 Rolling Archive

Production logs can't grow forever — `RollingFileAppender` rolls by time/size:

- One file per day, split when it exceeds 100MB.
- Keep 30 days of history.
- Auto-delete old files when total size exceeds 10GB.
- Historical files gzip-compressed.

```
logs/
├── app.log                       # Today
├── app.2024-01-13.0.log.gz       # Day before yesterday (compressed)
├── app.2024-01-14.0.log.gz
└── app.2024-01-14.1.log.gz       # Yesterday's second file (split after >100MB)
```

## 5. Performance and Async Logging

### 5.1 Parameterized Logging: Avoid String Concatenation

**Wrong way**:

```java
log.debug("User " + name + " logged in, id=" + id);   // Always concatenates!
```

Even when `DEBUG` is off, the string concatenation **still executes** — wasting CPU and memory.

**Right way**:

```java
log.debug("User {} logged in, id={}", name, id);
```

`{}` is a placeholder, replaced **only when the log is actually output** — when `DEBUG` is off the cost is almost zero. This is a key SLF4J design.

### 5.2 The Cost of Disabled Logging

```java
// Assuming DEBUG is off
log.debug("Complex compute: " + expensiveCompute());   // expensiveCompute still runs!

log.debug("Complex compute: {}", expensiveCompute());   // expensiveCompute still runs! (args must be computed)

// Use a guard to avoid expensive arguments
if (log.isDebugEnabled()) {
    log.debug("Complex compute: {}", expensiveCompute());
}
```

Note the second form — parameterized logging only saves on concatenation; **the argument itself still has to be computed**. If the argument is expensive (serializing a large object, querying a database), guard with `isDebugEnabled()`. But for simple arguments (primitive types, field access), no guard is needed — parameterized logging is enough.

### 5.3 Async Logging

Synchronous logging — writing to a file blocks business threads via IO. Under high concurrency, logging becomes the bottleneck. **Async logging** — the business thread drops the event into a queue, and a background thread writes in batches:

Logback Async Appender:

```xml
<appender name="ASYNC" class="ch.qos.logback.classic.AsyncAppender">
    <queueSize>4096</queueSize>
    <discardingThreshold>0</discardingThreshold>   <!-- 0 = don't discard any level -->
    <neverBlock>true</neverBlock>                   <!-- When queue is full, don't block business -->
    <appender-ref ref="FILE"/>
</appender>
```

- Business threads return as soon as they enqueue, without waiting for disk writes.
- Background thread writes in batches, 10–100x higher throughput.
- The trade-off — if the program crashes, unwritten logs in the queue are lost.

**Log4j 2**'s Async Logger is even more aggressive — it uses the Disruptor lock-free queue, more than 10x faster than Logback async. Choose Log4j 2 for extreme performance.

### 5.4 How to Log Exception Stacks

```java
// Right: the exception as the last argument, not inside a {}
try {
    ...
} catch (SQLException e) {
    log.error("Failed to query user: id={}", id, e);   // Exception auto-prints stack
}

// Wrong: e.getMessage() loses the stack
log.error("Query failed: {}", e.getMessage());   // No stack, how to locate?

// Wrong: string concatenation
log.error("Query failed: " + e);   // e.toString() has no stack
```

The exception object goes as the last argument of `log.error(msg, args, exception)` (without a `{}`); SLF4J automatically prints the full stack. `e.getMessage()` has only the message text, no stack — when something goes wrong you can't locate it.

## 6. MDC: Distributed Tracing

**MDC** (Mapped Diagnostic Context) — a thread-local logging context. Put a `traceId` at the start of a request, and subsequent logs automatically carry it:

```java
import org.slf4j.MDC;

// Request start
MDC.put("traceId", UUID.randomUUID().toString());
MDC.put("userId", "user42");

try {
    // Business code; logs everywhere will automatically carry traceId/userId
    log.info("Processing order");
} finally {
    MDC.clear();   // Clean up to avoid thread-pool leakage
}
```

Add `%X{traceId}` to the Pattern:

```
%d{HH:mm:ss} [%X{traceId}] %-5level %logger - %msg%n
```

Output:

```
14:30:25 [a1b2c3d4] INFO  c.e.OrderService - Processing order
```

MDC is the foundation of distributed tracing — all logs of the same request carry the same `traceId`, so you can one-click search the full call chain in ELK/Loki. Spring Cloud Sleuth/Micrometer Tracing auto-injects traceId based on MDC.

                } catch (InterruptedException ex) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
        }, "AsyncLogger");
        t.setDaemon(true);
        t.start();
    }

    @Override

        if (!queue.offer(e)) {
            // 队列满, 降级为同步 (生产框架是丢弃或阻塞)
            target.append(e);
        }

> **Key observations**:
> - **Parameterized logging is nearly zero-cost when the level is off** — this is SLF4J's key design, saving a million concatenations compared with string concatenation.
> - **The exception as the last argument auto-prints the stack** — don't use `e.getMessage()`, which loses the stack and makes locating impossible.
> - **MDC lets logs automatically carry `traceId`** — all logs of the same request can be chained together.
> - **Async logging submits 100 events almost instantly** — business threads don't wait for disk writes.

## 8. Logging Best Practices Checklist

1. **Use the SLF4J facade** — business code shouldn't depend directly on Logback/Log4j2; this makes swapping implementations easy.
2. **Always parameterize** — `log.info("x={}", x)`, never `log.info("x=" + x)`.
3. **Exception as the last argument** — `log.error("msg", e)`, auto-prints the stack.
4. **Be restrained with levels** — INFO should be precise, ERROR strict.
5. **Don't log heavily inside loops** — easily causes log explosion.
6. **Don't log sensitive information** — passwords, ID numbers, card numbers must never enter logs.
7. **Use async logging in production** — avoid IO blocking business.
8. **Logs should carry context** — `log.info("User {} logged in", name)` is far more useful than `log.info("Login succeeded")`.
9. **Roll file archives** — don't let logs blow up the disk.
10. **Don't throw after ERROR** — pick either log or throw, avoid duplicate records.
