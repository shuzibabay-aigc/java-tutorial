# Lambda Expressions

In mathematics, functions are first-class citizens — they can be assigned to variables, passed as arguments, and returned from other functions. But for a long time, Java functions could only "live inside" objects — to pass a "behavior," you had to write a class, instantiate it, and call its method. This "write a whole class just to pass one action" verbosity left Java playing catch-up in the wave of functional programming.

Java 8's Lambda expressions finally freed "functions" from the constraints of objects. They are as concise as mathematical formulas — `(x) -> x * x` is the square function. In this chapter, we'll thoroughly explain Lambda — from concepts to syntax, from comparisons to pitfalls.

## 1. Functional Programming Concepts

### 1.1 Functions as "First-Class Citizens"

In functional programming, functions — like integers and strings — are "first-class citizens":

- They can be assigned to variables.
- They can be passed as arguments.
- They can be returned as values.
- They can be stored in data structures.

Java previously couldn't do this — "behavior" had to be encapsulated in objects. To pass a "comparator" to `Collections.sort`, you had to write an anonymous inner class:

```java
Collections.sort(list, new Comparator<Integer>() {
    @Override
    public int compare(Integer a, Integer b) {
        return b - a;   // descending order
    }
});
```

To pass a single line of `b - a` logic, you wrote 5 lines of boilerplate — 4 of which were noise.

### 1.2 Lambda to the Rescue

Lambda lets you write "behavior" directly as an expression:

```java
Collections.sort(list, (a, b) -> b - a);
```

5 lines become 1, the noise disappears, and the logic is crystal clear. The essence of Lambda is: **treating functions as values to be passed around**.

## 2. Lambda Syntax

### 2.1 Basic Structure

The basic syntax of a Lambda:

```
(parameter list) -> expression or statement block
```

`-＞` is the Lambda operator — the left side is parameters, the right side is the function body.

Several forms:

```java
// 1. No parameters
() -> System.out.println("hello")

// 2. One parameter (parentheses optional)
x -> x * x
(x) -> x * x

// 3. Multiple parameters
(a, b) -> a + b

// 4. Expression body (auto-returns, no semicolon)
(x, y) -> x + y

// 5. Statement block (requires return and semicolons)
(x, y) -> {
    int sum = x + y;
    return sum;
}
```

### 2.2 Types Can Be Omitted

Parameter types can usually be omitted — the compiler infers them from the "target type":

```java
// Full version
Comparator<Integer> c = (Integer a, Integer b) -> a - b;
// Types omitted
Comparator<Integer> c = (a, b) -> a - b;
```

### 2.3 Single Parameter Can Omit Parentheses

When there's only one parameter, the parentheses can be dropped:

```java
Function<Integer, Integer> square = x -> x * x;
// equivalent to
Function<Integer, Integer> square = (x) -> x * x;
```

For zero or multiple parameters, the parentheses cannot be omitted.

### 2.4 Expression vs Statement Block

```java
// Expression form: concise, auto-returns
(x, y) -> x + y

// Statement block form: multiple statements allowed, manual return needed
(x, y) -> {
    System.out.println("calculating...");
    return x + y;
}
```

Use the expression form whenever possible — it's more concise. Use the statement block form only when you need multiple statements or side effects.

## 3. Lambda vs Anonymous Inner Classes

### 3.1 Conciseness Comparison

```java
// Anonymous inner class
Runnable r1 = new Runnable() {
    @Override
    public void run() {
        System.out.println("hello");
    }
};

// Lambda
Runnable r2 = () -> System.out.println("hello");
```

6 lines become 1.

### 3.2 Different `this` References

This is the most easily overlooked difference:

- **Anonymous inner class's `this`**: points to the anonymous class's own instance.
- **Lambda's `this`**: points to the "enclosing class" instance (the class where the Lambda is defined).

```java
public class Outer {
    String name = "Outer";

    void test() {
        // Anonymous inner class
        Runnable r1 = new Runnable() {
            @Override
            public void run() {
                System.out.println(this.getClass());   // Outer$1 (anonymous class)
                System.out.println(this);              // anonymous class instance
            }
        };

        // Lambda
        Runnable r2 = () -> {
            System.out.println(this.getClass());       // Outer
            System.out.println(this);                  // Outer instance
            System.out.println(name);                  // "Outer" (directly accesses enclosing field)
        };
    }
}
```

Lambda does not introduce a new `this` scope — it "transparently" uses the enclosing `this`. This makes accessing enclosing fields more natural in a Lambda.

### 3.3 Different Implementation Mechanisms

- **Anonymous inner class**: At compile time, generates a `.class` file (e.g., `Outer$1.class`), and instantiates it at runtime.
- **Lambda**: Java 8 uses the `invokedynamic` instruction to generate it dynamically — no extra `.class` file is produced, saving memory.

You can verify this with `javap`: anonymous inner classes have separate class files after compilation, Lambdas do not.

## 4. Functional Interfaces

### 4.1 What Is a Functional Interface

A Lambda cannot exist in a vacuum — it needs a "target type." That target is a **functional interface**:

> An interface with exactly one abstract method (default and static methods don't count).

```java
@FunctionalInterface
public interface Runnable {
    void run();   // the only abstract method
}

@FunctionalInterface
public interface Comparator<T> {
    int compare(T o1, T o2);   // the only abstract method
    // equals comes from Object, doesn't count
    // multiple default methods don't count
}
```

The `@FunctionalInterface` annotation is optional — when added, the compiler checks "whether there's exactly one abstract method," preventing accidental breakage. **Strongly recommended to add** — it's like a "contract stamp" for your interface.

### 4.2 Correspondence Between Lambda and Functional Interfaces

The Lambda's signature (parameter count, types, return value) must match the target interface's single abstract method:

```java
// Runnable.run() takes no params, returns nothing
Runnable r = () -> System.out.println("hi");

// Comparator.compare(T, T) takes two params, returns int
Comparator<Integer> c = (a, b) -> a - b;

// Callable<V>.call() takes no params, returns V
Callable<String> call = () -> "result";
```

The compiler infers the Lambda's parameter types and return type from the "target type" (the interface declared on the left) — this is "target typing."

### 4.3 Custom Functional Interfaces

Writing your own functional interface is simple:

```java
@FunctionalInterface
interface StringProcessor {
    String process(String input);
}

StringProcessor toUpper = s -> s.toUpperCase();
StringProcessor reverse = s -> new StringBuilder(s).reverse().toString();

System.out.println(toUpper.process("hello"));    // HELLO
System.out.println(reverse.process("hello"));    // olleh
```

However, Java already ships with a set of common functional interfaces (`Function`, `Predicate`, `Consumer`, `Supplier`, etc.), detailed in the next chapter. **Before writing your own, check whether the built-ins suffice** — avoid reinventing the wheel.

## 5. Type Inference

### 5.1 Target Typing

The compiler infers the Lambda's type from the context:

```java
// Same (x, y) -> x + y, different targets, different types
Comparator<Integer> c = (x, y) -> x - y;        // x, y are Integer, returns int
IntBinaryOperator op = (x, y) -> x + y;          // x, y are int, returns int
BiFunction<String, String, String> f = (x, y) -> x + y;  // x, y are String
```

### 5.2 Parameter Types Can Be Omitted

```java
// Explicit types
BinaryOperator<Integer> add = (Integer a, Integer b) -> a + b;
// Inferred types
BinaryOperator<Integer> add = (a, b) -> a + b;
```

Mixed style (one declared, one not) doesn't work: `(Integer a, b) -＞ ...` is a compile error — either declare all or none.

### 5.3 Return Type Inference

The return type is inferred from the function body's "return expression":

```java
// Infers return type String
Supplier<String> s = () -> "hello";

// Infers return type int
IntSupplier is = () -> 42;
```

If the function body has multiple returns, the types must be consistent.

## 6. Variable Capture

### 6.1 Lambdas Can Access Enclosing Variables

A Lambda can access local variables of its enclosing method, fields of the enclosing class, and method parameters:

```java
public class Example {
    int field = 10;   // enclosing field

    void method(int param) {
        int local = 20;   // local variable

        Runnable r = () -> {
            System.out.println(field);     // ✅ access field
            System.out.println(param);     // ✅ access parameter
            System.out.println(local);     // ✅ access local variable
        };
    }
}
```

### 6.2 The effectively final Restriction

**Key rule**: Local variables captured by a Lambda must be effectively final — meaning they are not modified after declaration.

```java
int x = 10;
Runnable r = () -> System.out.println(x);   // ✅ x not modified, effectively final

int y = 10;
y = 20;   // modified!
Runnable r2 = () -> System.out.println(y);  // ❌ compile error
```

### 6.3 Why This Restriction Exists

This restriction isn't Java being deliberately difficult — it's based on **concurrency safety** and **semantic clarity**:

1. **Concurrency issues**: A Lambda may execute in another thread. If it could modify captured local variables, concurrent modifications across threads would cause data races. Java doesn't have C#'s `ref` parameter mechanism, so it can't safely "pass by reference."

2. **Consistency**: Local variables live on the method stack, and the stack frame is destroyed when the method returns. If the Lambda executes after the method returns (e.g., an async callback), the "local variable" it accesses is actually a **copy**. If modification were allowed, inconsistency between the copy and the original would be confusing.

3. **State management**: Functional programming favors "no side effects." Allowing modification of captured variables introduces implicit state, violating this principle.

Note: **Fields don't have this restriction** — fields are object state, stored on the heap, and the Lambda accesses them by reference, not by copy. But you still need to ensure thread safety yourself when modifying fields across threads.

### 6.4 Workarounds for the Restriction

If you need to "accumulate" a value inside a Lambda, use an array or an Atomic class to "wrap" it:

```java
// ❌ Doesn't work
int sum = 0;
list.forEach(x -> sum += x);   // compile error

// ✅ Use an array
int[] sum = {0};
list.forEach(x -> sum[0] += x);

// ✅ Use Atomic (thread-safe)
AtomicInteger sum = new AtomicInteger();
list.forEach(x -> sum.addAndGet(x));

// ✅ Better approach: use Stream
int sum = list.stream().mapToInt(Integer::intValue).sum();
```

The last approach is most recommended — it has no side effects and is true functional style.

## 7. Common Lambda Use Cases

### 7.1 Collection Sorting

```java
list.sort((a, b) -> a.length() - b.length());   // sort by length ascending
list.sort(Comparator.comparingInt(String::length));   // more idiomatic
```

### 7.2 Collection Iteration

```java
list.forEach(s -> System.out.println(s));
list.forEach(System.out::println);   // method reference is more concise
```

### 7.3 Collection Removal

```java
list.removeIf(s -> s.isEmpty());   // remove empty strings
```

### 7.4 Stream Operations

```java
list.stream()
    .filter(s -> s.length() > 3)
    .map(String::toUpperCase)
    .forEach(System.out::println);
```

### 7.5 Thread Creation

```java
new Thread(() -> {
    System.out.println("running in a child thread");
}).start();
```

### 7.6 Event Listeners (GUI, Callbacks)

```java
button.addActionListener(e -> handleClick(e));
```

## 8. Lambda Limitations

Lambda isn't a silver bullet — it has several limitations:

1. **Only works with functional interfaces**: To pass to an interface with "multiple abstract methods" (like `List`), you still need anonymous inner classes.
2. **Cannot have state fields**: A Lambda can't define fields like an anonymous inner class can. Use anonymous inner classes if you need to hold state.
3. **Cannot recurse itself**: A Lambda has no name, so it can't call itself directly (you'd need method references or assign it to a variable for indirect recursion).
4. **Cannot throw checked exceptions**: Unless the functional interface's abstract method declares that exception.

```java
// ❌ Cannot throw checked exception (Runnable.run doesn't declare exceptions)
Runnable r = () -> { throw new IOException(); };

// ✅ Throwing runtime exceptions is fine
Runnable r2 = () -> { throw new RuntimeException(); };

// ✅ Works only if the interface method declares the checked exception
interface Task { void run() throws IOException; }
Task t = () -> { throw new IOException(); };
```

## Conclusion: Freedom of Functions

Lambda expressions are Java's first — and most important — step toward functional programming. They free "behavior" from the heavy shell of "objects," letting code return to "logic" itself — a single line of `(a, b) -＞ a + b` is far cleaner than an anonymous inner class.

But behind Lambda's simplicity lie rules: it needs a functional interface as its target type, and captured local variables must be effectively final. Understanding these rules lets you write Lambdas that are both elegant and correct.

In the next chapter, we'll look at Java's built-in functional interfaces — `Function`, `Predicate`, `Consumer`, `Supplier`, and more. They are the "stage" for Lambdas and the cornerstone of the Stream API.
