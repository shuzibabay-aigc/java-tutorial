# Design Patterns

Welcome to Stage 14 — **Advanced Topics and Distributed Systems**. In previous stages we covered language mechanics, engineering practices, and web frameworks. This stage raises the bar — design patterns, data structures and algorithms, Redis, message queues, microservices, and Docker — taking you from "knows how to use frameworks" to "can design architectures." First stop: **Design Patterns**.

A design pattern is a distillation of hard-won experience — what the "standard answer" looks like when facing a specific problem. The 23 patterns cataloged by the GoF (Gang of Four) are the classics. In this chapter we'll examine the six design principles, the three pattern categories, and concrete examples living inside the Java standard library.

## I. The Six Design Principles

The bedrock of design patterns is **design principles** — follow the principles and good code follows; patterns are merely principles applied in concrete form.

### 1.1 The SOLID Five

| Principle | Full Name | Meaning |
| --- | --- | --- |
| **S** — Single Responsibility | Single Responsibility | A class has only one reason to change |
| **O** — Open-Closed | Open-Closed | Open for extension, closed for modification |
| **L** — Liskov Substitution | Liskov Substitution | Subtypes must be substitutable for their base types |
| **I** — Interface Segregation | Interface Segregation | Keep interfaces small and focused; no fat interfaces |
| **D** — Dependency Inversion | Dependency Inversion | Depend on abstractions, not concretions |

**Single Responsibility (SRP)** — one class, one job. `UserService` shouldn't handle both authentication and sending emails — split it into `UserService` and `EmailService`.

**Open-Closed Principle (OCP)** — add new features without touching old code. The Strategy pattern is a perfect example: adding a new strategy means adding a new class, not modifying `Context`.

**Liskov Substitution (LSP)** — a subclass must not violate its parent's contract. If the parent's `add(a, b)` returns `a+b` and a subclass overrides it to return `a-b`, LSP is broken — anywhere the parent is expected, swapping in the child will misbehave.

**Interface Segregation (ISP)** — keep interfaces small. If a `UserService` interface bundles both read and write methods, a read-only client is forced to depend on writes it doesn't need — split it into `UserReader` and `UserWriter`.

**Dependency Inversion (DIP)** — high-level modules should not depend on low-level modules; both should depend on abstractions. `UserService` (high level) should not depend directly on `UserRepositoryImpl` (low level), but on the `UserRepository` interface — this is the very idea behind Spring IoC.

### 1.2 The Law of Demeter (LoD)

**The Law of Demeter** — the principle of least knowledge. An object should talk only to its "friends," not to "strangers."

```java
// Violates LoD: talking to a stranger's stranger
order.getCustomer().getAddress().getCity();

// Follows LoD: ask a direct friend for help
order.getCustomerCity();
```

Don't dig deep into objects you barely know — only call methods on your direct dependencies. This keeps coupling low.

## II. Creational Patterns

Creational patterns are about "how to create objects" — decoupling creation from usage.

### 2.1 Singleton

Guarantees a class has exactly one instance. In the Java standard library, `Runtime.getRuntime()` is a singleton.

```java
// Enum singleton (best practice: reflection-safe and serialization-safe)
public enum Singleton {
    INSTANCE;
    public void doSomething() { ... }
}

// Double-checked locking (the classic form)
public class Singleton {
    private static volatile Singleton instance;
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

`volatile` is mandatory — it prevents instruction reordering from leaking a half-initialized object.

### 2.2 Factory Method

Defines an interface for creating objects but lets subclasses decide which class to instantiate.

```java
interface Product { void use(); }
class ProductA implements Product { public void use() { System.out.println("用 A"); } }
class ProductB implements Product { public void use() { System.out.println("用 B"); } }

abstract class Factory {
    abstract Product create();
}
class FactoryA extends Factory { Product create() { return new ProductA(); } }
class FactoryB extends Factory { Product create() { return new ProductB(); } }

// Usage
Factory f = new FactoryA();
Product p = f.create();   // ProductA
```

Spring's `BeanFactory` is the factory pattern in action — `getBean(name)` returns a bean instance.

### 2.3 Builder

Separates the construction of a complex object from its representation — building it step by step.

```java
public class User {
    private final String name;
    private final int age;
    private final String email;
    private final String address;

    private User(Builder b) {
        this.name = b.name; this.age = b.age;
        this.email = b.email; this.address = b.address;
    }

    public static class Builder {
        private String name; private int age;
        private String email; private String address;

        public Builder name(String n) { this.name = n; return this; }
        public Builder age(int a) { this.age = a; return this; }
        public Builder email(String e) { this.email = e; return this; }
        public Builder address(String a) { this.address = a; return this; }
        public User build() { return new User(this); }
    }
}

// Usage
User u = new User.Builder()
    .name("张三").age(20).email("a@b.com").build();
```

Lombok's `@Builder` generates all of this for you. `StringBuilder` is also a Builder pattern.

### 2.4 Prototype

Creates new objects by cloning an existing one — implement the `Cloneable` interface.

```java
public class Prototype implements Cloneable {
    private String data;
    public Prototype clone() {
        try { return (Prototype) super.clone(); }
        catch (CloneNotSupportedException e) { throw new RuntimeException(e); }
    }
}
```

`Object.clone()` performs a shallow copy — primitives and references are copied, but the objects those references point to are not. Deep copies require recursive cloning or serialization.

## III. Structural Patterns

Structural patterns are about "how classes and objects are composed" — adapting, decorating, proxying.

### 3.1 Adapter

Lets classes with incompatible interfaces work together — converts one interface into another.

```java
// Old interface
interface OldPrinter { void printOld(String s); }
class OldPrinterImpl implements OldPrinter {
    public void printOld(String s) { System.out.println("旧打印: " + s); }
}

// New interface
interface NewPrinter { void print(String s); }

// Adapter: wraps the old interface as the new one
class PrinterAdapter implements NewPrinter {
    private OldPrinter old;
    public PrinterAdapter(OldPrinter old) { this.old = old; }
    public void print(String s) { old.printOld(s); }
}
```

JDK's `InputStreamReader` is an adapter — it adapts an `InputStream` (byte stream) into a `Reader` (character stream).

### 3.2 Decorator

Dynamically adds behavior to an object — more flexible than inheritance.

```java
interface Coffee { double cost(); }
class SimpleCoffee implements Coffee {
    public double cost() { return 10; }
}

// Decorator: add milk
class MilkDecorator implements Coffee {
    private Coffee coffee;
    public MilkDecorator(Coffee c) { this.coffee = c; }
    public double cost() { return coffee.cost() + 2; }
}
// Decorator: add sugar
class SugarDecorator implements Coffee {
    private Coffee coffee;
    public SugarDecorator(Coffee c) { this.coffee = c; }
    public double cost() { return coffee.cost() + 1; }
}

// Usage
Coffee c = new SugarDecorator(new MilkDecorator(new SimpleCoffee()));
c.cost();   // 10 + 2 + 1 = 13
```

JDK's `BufferedReader` and `DataInputStream` are decorators — they add buffering and typed read/write on top of `InputStream`.

### 3.3 Proxy

Wraps an object with a proxy to control access. The dynamic proxies from Stage 10 are an application of this.

```java
interface Service { void execute(); }
class RealService implements Service {
    public void execute() { System.out.println("执行"); }
}
class ProxyService implements Service {
    private RealService real;
    public void execute() {
        if (real == null) real = new RealService();
        System.out.println("[前置]");
        real.execute();
        System.out.println("[后置]");
    }
}
```

### 3.4 Facade

Gives a complex subsystem a single unified entry point — simplifying the call.

```java
class CPU { void start() { System.out.println("CPU 启动"); } }
class Memory { void load() { System.out.println("内存加载"); } }
class Disk { void read() { System.out.println("硬盘读取"); } }

class ComputerFacade {
    private CPU cpu = new CPU();
    private Memory memory = new Memory();
    private Disk disk = new Disk();

    public void start() {
        cpu.start();
        memory.load();
        disk.read();
        System.out.println("电脑启动完成");
    }
}

// Usage
new ComputerFacade().start();   // Done in one line, no need to care about internals
```

Spring's `JdbcTemplate` is a Facade — it distills JDBC's complex operations down to a handful of methods like `query` and `update`.

### 3.5 Composite

A tree structure — treats individual objects and compositions uniformly.

```java
interface Component { void display(); }
class Leaf implements Component {
    private String name;
    public Leaf(String n) { name = n; }
    public void display() { System.out.println("叶子: " + name); }
}
class Composite implements Component {
    private List<Component> children = new ArrayList<>();
    private String name;
    public Composite(String n) { name = n; }
    public void add(Component c) { children.add(c); }
    public void display() {
        System.out.println("分支: " + name);
        for (Component c : children) c.display();
    }
}
```

Swing's `Container`/`Component` is the Composite pattern — a container can hold both components and other containers.

## IV. Behavioral Patterns

Behavioral patterns are about "how objects communicate and how responsibilities are assigned."

### 4.1 Strategy

Defines a family of algorithms, encapsulates them, and makes them interchangeable — to avoid a thicket of if-else.

```java
interface PayStrategy { void pay(double amount); }
class WechatPay implements PayStrategy {
    public void pay(double a) { System.out.println("微信支付 " + a); }
}
class AliPay implements PayStrategy {
    public void pay(double a) { System.out.println("支付宝 " + a); }
}

class PayContext {
    private PayStrategy strategy;
    public PayContext(PayStrategy s) { this.strategy = s; }
    public void checkout(double amount) { strategy.pay(amount); }
}

// Usage
new PayContext(new WechatPay()).checkout(99.5);
new PayContext(new AliPay()).checkout(99.5);
```

Adding a new payment method means adding a new `PayStrategy` implementation — `PayContext` stays untouched. That is the Open-Closed Principle in action.

### 4.2 Observer

A one-to-many dependency — when one object changes state, all its dependents are notified.

```java
interface Observer { void update(String event); }
class EmailNotifier implements Observer {
    public void update(String e) { System.out.println("邮件通知: " + e); }
}
class SmsNotifier implements Observer {
    public void update(String e) { System.out.println("短信通知: " + e); }
}

class Subject {
    private List<Observer> observers = new ArrayList<>();
    public void subscribe(Observer o) { observers.add(o); }
    public void notifyAll(String event) {
        for (Observer o : observers) o.update(event);
    }
}
```

Spring's `ApplicationEvent`/`ApplicationListener` is the Observer pattern.

### 4.3 Template Method

Defines the skeleton of an algorithm, letting subclasses override specific steps.

```java
abstract class Game {
    public final void play() {   // Template method, skeleton is fixed
        init();
        start();
        end();
    }
    protected abstract void init();
    protected abstract void start();
    protected void end() { System.out.println("游戏结束"); }
}
class Chess extends Game {
    protected void init() { System.out.println("摆棋盘"); }
    protected void start() { System.out.println("下棋"); }
}
```

`AbstractList`'s `indexOf` is a template method — it calls the `size()`/`get()` that subclasses implement.

### 4.4 Chain of Responsibility

A request travels along a chain; each handler decides to handle it or pass it to the next.

```java
abstract class Handler {
    protected Handler next;
    public Handler setNext(Handler n) { this.next = n; return n; }
    public abstract void handle(String request);
}
class AuthHandler extends Handler {
    public void handle(String r) {
        if (r.contains("auth")) System.out.println("认证通过");
        else if (next != null) next.handle(r);
    }
}
class LogHandler extends Handler {
    public void handle(String r) {
        System.out.println("记录日志: " + r);
        if (next != null) next.handle(r);
    }
}

// Usage
Handler chain = new AuthHandler();
chain.setNext(new LogHandler());
chain.handle("auth");
```

Spring's `FilterChain` and `Interceptor` chains are both Chain of Responsibility.

### 4.5 State

When an object's state changes, its behavior changes too — encapsulate each state as a class.

```java
interface State { void handle(); }
class IdleState implements State { public void handle() { System.out.println("空闲"); } }
class BusyState implements State { public void handle() { System.out.println("忙"); } }

class Worker {
    private State state;
    public void setState(State s) { this.state = s; }
    public void work() { state.handle(); }
}
```

Design patterns were never "invented" — they are "summaries" of excellent code. The JDK is full of them.

> - **Strategy + Factory eliminate if-else** — adding a new payment method only requires registration; the business code stays untouched. That's the Open-Closed Principle.

## VII. Chapter Summary

| Category | Patterns |
| --- | --- |
| 6 Principles | SRP / OCP / LSP / ISP / DIP / LoD |
| Creational | Singleton / Factory / Builder / Prototype |
| Structural | Adapter / Decorator / Proxy / Facade / Composite |
| Behavioral | Strategy / Observer / Template Method / Chain of Responsibility / State |

**Memory hooks**:

- **SOLID five** — Single, Open, Liskov, Interface, Dependency.
- **Strategy kills if-else** — one class per branch, the factory looks them up.
- **Decorator beats inheritance** — composition over inheritance.
- **Observer is one-to-many** — events decouple.
- **Template Method sets the skeleton** — subclasses fill in the details.
- **Chain of Responsibility links them up** — a request walks through node by node.

## Conclusion: Patterns Are a Means, Not an End

Design patterns are crystallized experience — but don't abuse them. Forcing a strategy pattern onto a three-line if-else is over-engineering. The essence of a pattern is "solving a specific problem" — reach for a pattern when you meet the problem; don't go hunting for problems to fit your patterns.
