# A Quick Tour of Other Modern Features

This chapter is Stage 8's finale — the previous four chapters covered the "heavyweight" features: the module system, Records, Sealed Classes, and Pattern Matching. This chapter knocks out the remaining "small but beautiful" features in one go. None is as revolutionary as the three-piece set, but together they make writing Java much more pleasant.

This chapter quickly covers six features:

1. **Interface private methods** (Java 9+)
2. **Stream enhancements** (Java 9+ / 16+)
3. **HttpClient** (Java 11+)
4. **Text Blocks** (Java 15+)
5. **Sequenced Collections** (Java 21+)
6. **Foreign Function & Memory API** (Java 22+, finalized)

## 1. Interface Private Methods (Java 9+)

### 1.1 The Pain Point: Code Reuse in Interfaces

Java 8 introduced `default` and `static` methods in interfaces — interfaces could finally carry "implementation." But a problem quickly emerged: how do multiple default methods share code?

```java
interface Logger {
    default void logInfo(String msg) {
        // Want to reuse this formatting logic
        System.out.println("[INFO] " + msg);
    }
    default void logError(String msg) {
        // Write the formatting again?
        System.out.println("[ERROR] " + msg);
    }
}
```

Java 8 had no way out — you had to write the "shared logic" as a `default` method too, but then it was **exposed to the interface's users**. Users could see "internal methods" like `logInternal`, breaking abstraction.

### 1.2 Java 9's Solution: Interface Private Methods

Java 9 (JEP 213) allows `private` methods in interfaces — callable only by default/static methods within the same interface:

```java
interface Logger {
    default void logInfo(String msg) {
        log("INFO", msg);   // call private method
    }
    default void logError(String msg) {
        log("ERROR", msg);
    }
    default void logWarn(String msg) {
        log("WARN", msg);
    }

    // Private method, reuses logic, not exposed to implementers
    private void log(String level, String msg) {
        System.out.println("[" + level + "] " + msg + " @" + System.currentTimeMillis());
    }

    // Can also be static private
    private static String format(String level, String msg) {
        return "[" + level + "] " + msg;
    }
}
```

Characteristics:

- **`private` methods**: instance methods, called by default methods.
- **`private static` methods**: static methods, called by static/default methods.
- **Cannot be abstract** — private methods must have an implementation.
- **Not inherited by implementers** — visible only inside the interface.

A small feature, but it solves interface code reuse's "last mile."

## 2. Stream Enhancements (Java 9+ / 16+)

Java 8 introduced the Stream API, and later versions kept "patching" it. The most commonly used enhancements:

### 2.1 takeWhile / dropWhile (Java 9+)

`takeWhile`: takes the prefix that satisfies the predicate, stopping at the first element that doesn't.

`dropWhile`: skips the prefix that satisfies the predicate, starting to take from the first element that doesn't.

```java
Stream.of(1, 2, 3, 4, 5, 2, 1)
    .takeWhile(n -> n < 4)   // take 1, 2, 3, stop at 4
    .collect(Collectors.toList());   // [1, 2, 3]

Stream.of(1, 2, 3, 4, 5, 2, 1)
    .dropWhile(n -> n < 4)   // skip 1, 2, 3, start from 4
    .collect(Collectors.toList());   // [4, 5, 2, 1]
```

**Difference from `filter`** — `filter` is "filter the whole stream," `takeWhile`/`dropWhile` are "prefix operations." Especially useful for **ordered streams segmented by some condition**: log pagination by time, sequence segmentation by threshold.

### 2.2 ofNullable (Java 9+)

`Stream.ofNullable(null)` returns an empty stream, no NPE:

```java
Stream.ofNullable(null).count();   // 0
Stream.ofNullable("hello").count();   // 1

// Pair with flatMap to handle possibly null elements
Stream.of("a", null, "b", null, "c")
    .flatMap(Stream::ofNullable)   // filter out nulls
    .collect(Collectors.toList());   // [a, b, c]
```

### 2.3 Stream.toList() (Java 16+)

Before Java 16, collecting a Stream into a List required `.collect(Collectors.toList())` — long and error-prone. Java 16 added `Stream.toList()` for direct collection:

```java
// Old style
List<String> list = stream.collect(Collectors.toList());

// New style
List<String> list = stream.toList();
```

**Note**: `toList()` returns an **immutable List** — you can't `add`/`remove`. For a mutable List, use `collect(Collectors.toCollection(ArrayList::new))`.

```java
List<Integer> nums = Stream.of(1, 2, 3).toList();
nums.add(4);   // throws UnsupportedOperationException
```

### 2.4 Other Minor Enhancements

- `Stream.iterate(seed, hasNext, next)` — iterate with a termination condition, like a for loop.
- `IntStream.takeWhile/dropWhile` — primitive streams also support them.
- `Collectors.teeing` — dual-downstream collector, computing two stats at once (e.g., sum and count for an average).

```java
// Java 9: iterate with termination condition
Stream.iterate(1, n -> n <= 100, n -> n + 1)
    .forEach(System.out::println);   // equivalent to for (int i=1; i<=100; i++)

// Java 12+: teeing computes sum and count at once
double avg = Stream.of(1, 2, 3, 4, 5)
    .collect(Collectors.teeing(
        Collectors.summingDouble(i -> i),
        Collectors.counting(),
        (sum, count) -> sum / count
    ));   // 3.0
```

## 3. HttpClient (Java 11+)

Java 9 introduced a new HTTP Client API (`java.net.http`), finalized in Java 11 (JEP 321). It replaces the aging `HttpURLConnection` — supporting HTTP/2, WebSocket, sync/async, and connection pooling.

### 3.1 Basic Usage

```java
import java.net.URI;
import java.net.http.*;
import java.time.Duration;

HttpClient client = HttpClient.newBuilder()
    .version(HttpClient.Version.HTTP_2)        // default HTTP/2, auto-fallback to HTTP/1.1
    .connectTimeout(Duration.ofSeconds(10))
    .build();

HttpRequest request = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/users"))
    .timeout(Duration.ofSeconds(30))
    .header("Accept", "application/json")
    .GET()
    .build();

// Synchronous
HttpResponse<String> response = client.send(request, HttpResponse.BodyHandlers.ofString());
System.out.println(response.statusCode());
System.out.println(response.body());

// Asynchronous
client.sendAsync(request, HttpResponse.BodyHandlers.ofString())
    .thenApply(HttpResponse::body)
    .thenAccept(System.out::println)
    .join();   // wait for completion
```

### 3.2 Advantage Comparison

| Dimension | HttpURLConnection | HttpClient |
| --- | --- | --- |
| HTTP/2 | Not supported | Native support |
| Async | You spin your own threads | Built-in `sendAsync` |
| Connection pool | None (new each time) | Built-in pool |
| API design | Ancient, verbose config | Builder pattern, clear |
| WebSocket | Not supported | Native support |
| HTTPS | Verbose config | Supported by default |

### 3.3 POST JSON

```java
String json = "{\"name\":\"Alice\",\"age\":30}";
HttpRequest postReq = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/users"))
    .header("Content-Type", "application/json")
    .POST(HttpRequest.BodyPublishers.ofString(json))
    .build();
```

### 3.4 Note

HttpClient itself is **thread-safe** and reusable — typically an application creates one client and shares it globally. Each request creates an `HttpRequest` object (immutable, reusable).

## 4. Text Blocks (Java 15+)

### 4.1 The Pain Point: "Escape Hell" in Strings

Writing SQL, JSON, HTML — Java string escaping drives you crazy:

```java
String json = "{\\n" +
    "  \\"name\\": \\"Alice\\",\\n" +
    "  \\"age\\": 30\\n" +
    "}";
```

Every line `+` concatenated, every `"` escaped to `\"` — 10 lines of JSON looks like hieroglyphics.

### 4.2 Text Block Syntax

Java 15 (JEP 378) introduced **Text Blocks** — multi-line strings wrapped in three double quotes `"""`:

```java
String json = """
        {
          "name": "Alice",
          "age": 30
        }
        """;
```

Characteristics:

- **Multi-line** — line breaks directly, no `\n`.
- **No quote escaping** — write `"` directly.
- **Indentation control** — the compiler automatically removes the "common minimal indent," aligning the code.
- **Closing `"""`** on its own line marks the end of the string.

### 4.3 Indentation Rules

Text block indentation is "smart" — based on the position of the **closing `"""`**:

```java
String s = """
        hello
        world
        """;   // closing """ here
// s = "hello\\nworld\\n"  (indentation removed)
```

If the closing `"""` is further left, more indentation is preserved:

```java
String s = """
            hello
            world
        """;   // closing """ further left
// s = "    hello\\n    world\\n"  (4 spaces preserved)
```

### 4.4 Escaping and Interpolation

Inside text blocks you can still use `\s` (preserve trailing spaces), `\<newline>` (line continuation):

```java
String sql = """
        SELECT id, name \\
               age \\
        FROM   users \\
        WHERE  age > 18
        """;
// Actually "SELECT id, name age FROM users WHERE age > 18" (one line)

String colors = """
        red   \\s
        green \\s
        blue  \\s
        """;   // \\s preserves trailing spaces, otherwise the compiler strips them
```

Java has no built-in string interpolation (like `${var}`), but you can use `String.format` or `formatted`:

```java
String name = "Alice";
int age = 30;
String json = """
        {
          "name": "%s",
          "age": %d
        }
        """.formatted(name, age);
```

## 5. Sequenced Collections (Java 21+)

### 5.1 The Pain Point: Getting the "Last Element" Is That Hard?

Before Java 21, different collections had completely inconsistent APIs for "getting the last element":

```java
List<Integer> list = List.of(1, 2, 3);
list.get(list.size() - 1);   // List's last

Deque<Integer> deque = new ArrayDeque<>(list);
deque.getLast();   // Deque's last

SortedSet<Integer> set = new TreeSet<>(list);
set.last();   // SortedSet's last

// Set has "no order", getting last is meaningless (but LinkedHashSet has order...)
```

Each collection type had its own API, impossible to remember. `Map` was worse — getting the last entry required `map.entrySet().iterator()` and iterating `next` to the end.

### 5.2 The Sequenced Collections Interface

Java 21 (JEP 431) introduced three new interfaces to unify "ordered collections":

- `SequencedCollection＜E＞` — an ordered Collection
- `SequencedSet＜E＞` — an ordered Set
- `SequencedMap＜K,V＞` — an ordered Map

New methods:

```java
interface SequencedCollection<E> extends Collection<E> {
    SequencedCollection<E> reversed();   // reverse
    void addFirst(E e);                  // add to head
    void addLast(E e);                   // add to tail
    E getFirst();                        // get first
    E getLast();                         // get last
    E removeFirst();                     // remove first
    E removeLast();                      // remove last
}
```

```java
List<Integer> list = new ArrayList<>(List.of(1, 2, 3, 4, 5));
list.getFirst();      // 1
list.getLast();       // 5
list.addFirst(0);     // [0, 1, 2, 3, 4, 5]
list.reversed();      // [5, 4, 3, 2, 1, 0]

LinkedHashSet<Integer> set = new LinkedHashSet<>(List.of(1, 2, 3));
set.getFirst();       // 1
set.getLast();        // 3   (impossible before!)

LinkedHashMap<String, Integer> map = new LinkedHashMap<>();
map.put("a", 1); map.put("b", 2); map.put("c", 3);
map.firstEntry();     // a=1
map.lastEntry();      // c=3   (impossible before!)
map.pollLastEntry();  // removes and returns c=3
map.reversed();       // reversed Map view
```

Finally — all "ordered collections" share one API. Readability and portability take a big leap.

### 5.3 Which Collections Implement It

- `List`, `Deque`, `Queue` → implement `SequencedCollection`
- `LinkedHashSet` → implements `SequencedSet`
- `LinkedHashMap` → implements `SequencedMap`
- `TreeSet` / `TreeMap` also implement (by comparison order)
- `HashSet` / `HashMap` do not (no order)

## 6. Foreign Function & Memory API (Java 22+)

### 6.1 The Pain Point: JNI's Torment

Java calling C/C++ libraries always used JNI (Java Native Interface) — but it has big problems:

- **Writing C bridge code** — you have to write `.h` headers, `.c` implementations, compile to `.so`/`.dll`, complex deployment.
- **Memory-unsafe** — C pointers, memory allocations are outside the JVM's reach, easily causing segfaults.
- **Performance overhead** — JNI calls have context-switch overhead.
- **Can't call system functions** — calling `printf`, `getpid` and other libc functions also requires C wrappers.

### 6.2 What the FFM API Is

**The Foreign Function & Memory API (FFM)** is a feature finalized in Java 22 (JEP 454), letting Java **directly call** C/C++ library functions and **directly manipulate** off-heap memory, **without writing JNI bridge code**.

Core APIs:

- `Linker` — platform linker, finds C functions.
- `SymbolLookup` — finds symbols in `.so`/`.dll`.
- `MemorySegment` — off-heap memory segment, replaces `sun.misc.Unsafe` and `ByteBuffer`.
- `Arena` — memory allocator, manages lifecycle.

### 6.3 Calling a C Function Example

```java
import java.lang.foreign.*;
import java.lang.invoke.MethodHandle;

Linker linker = Linker.nativeLinker();
SymbolLookup stdlib = linker.defaultLookup();

// Find C's printf function
MethodHandle printf = stdlib.find("printf")
    .orElseThrow();

// Call
try (Arena arena = Arena.ofConfined()) {
    MemorySegment format = arena.allocateUtf8String("Hello %s!\\n");
    MemorySegment arg = arena.allocateUtf8String("World");
    printf.invoke(format, arg);   // outputs "Hello World!"
}
```

Not a line of C code, yet directly calling `printf` — unimaginable in the JNI era. Memory managed by `Arena.ofConfined()` is automatically released when try-with-resources ends, **memory-safe**.

### 6.4 Performance and Use Cases

FFM's design goal is **performance close to JNI, memory safer**. It replaces three old APIs:

- `sun.misc.Unsafe` (internal API, slated for removal)
- `java.nio.ByteBuffer`'s direct memory (limited API)
- JNI (complex development)

Suitable scenarios: high-performance computing, calling native libraries (OpenSSL, SQLite, CUDA), game engines, database drivers. From Java 22 onward, official database drivers and TLS libraries are being rewritten with FFM.

## 8. Other Notable Features

Beyond the six big ones, here are some other modern features worth knowing:

- **var local variable type inference** (Java 10+) — `var list = new ArrayList＜String＞()`.
- **switch expressions** (Java 14+) — `switch` can be an expression returning a value, with the `yield` keyword.
- **`String.isBlank()`/`strip()`/`repeat()`/`lines()`** (Java 11+) — string enhancements.
- **`Files.readString`/`writeString`** (Java 11+) — read/write text files in one line.
- **`Optional.isEmpty()`** (Java 11+) — symmetric to `!isPresent()`.
- **`String.format`'s instance version `formatted`** (Java 15+) — `"%s".formatted(name)`.
- **Record Patterns nesting** (Java 21) — covered in the previous chapter.
- **Virtual Threads** (Java 21) — covered in Stage 7.
- **Generational ZGC** (Java 21) — covered in the next stage.
- **Unnamed Patterns/Variables** (Java 22) — `case Point(_, _)`, `var _ = ...`.
- **Statements Before super/this** (Java 22) — statements before super in constructors.

Java's six-month release cadence lets features "run in small steps" — every release ships a few small features, accumulating into qualitative change.

- **toList is immutable** — for mutable, use `toCollection(ArrayList::new)`.
- **HttpClient three-piece set** — HttpClient, HttpRequest, HttpResponse, Builder pattern.
- **Text Blocks triple-quote** — indentation depends on the closing `"""`'s position.
- **Sequenced Collections** — all ordered collections' `getFirst`/`getLast`/`reversed` unified.
- **FFM** — only finalized in Java 22+, call C without JNI.
