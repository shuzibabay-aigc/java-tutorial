# Spring Core

**Spring** is the de facto standard framework for Java back-ends — released by Rod Johnson in 2003, it has dominated enterprise Java development ever since. It solves three core problems:

- **IoC/DI** — Object creation and dependency management are handed off to the container; no more manual `new`.
- **AOP** — Cross-cutting concerns (transactions, logging, permissions) are separated from business code.
- **Template encapsulation** — Boilerplate code for JDBC/transactions/ORM is wrapped up so business logic takes center stage.

This chapter examines Spring's two great pillars — IoC and AOP — and the principles behind them.

## 1. The Spring Ecosystem at a Glance

Spring is not a single framework; it is an ecosystem:

| Project | Purpose |
| --- | --- |
| **Spring Framework** | The core (IoC/AOP/transactions); the foundation of every other project |
| **Spring Boot** | Auto-configuration + starter dependencies; ready out of the box |
| **Spring MVC** | Web MVC framework (Servlet-based) |
| **Spring WebFlux** | Reactive web framework (Reactor-based) |
| **Spring Data** | Data access abstraction (JPA/Redis/MongoDB) |
| **Spring Security** | Authentication and authorization |
| **Spring Cloud** | Microservices toolkit (gateway/registry/config) |
| **Spring Batch** | Batch processing |
| **Spring Integration** | EIP integration patterns |

This chapter focuses on the Spring Framework core — IoC and AOP.

## 2. IoC and DI

### 2.1 Traditional Approach vs IoC

In traditional development, objects `new` their own dependencies:

```java
public class UserService {
    private UserRepository repo = new UserRepositoryImpl();   // manual new
    private EmailService email = new EmailServiceImpl();
    // Tightly coupled: switching implementations requires code changes, hard to test
}
```

The problems:

- **Tight coupling** — `UserService` hard-codes `UserRepositoryImpl`; switching implementations means changing code.
- **Hard to test** — You cannot mock `UserRepository` in unit tests.
- **Chaotic lifecycle management** — Each `new` is self-managed; objects proliferate.

**IoC (Inversion of Control)** — Reverse the control over "object creation" and hand it to the container:

```java
public class UserService {
    private final UserRepository repo;    // declare only, do not new
    private final EmailService email;

    // Constructor injection: the container injects dependencies
    public UserService(UserRepository repo, EmailService email) {
        this.repo = repo;
        this.email = email;
    }
}
```

The container is responsible for `new`-ing objects, injecting dependencies, and managing lifecycles — that is "inversion of control." **DI (Dependency Injection)** is the implementation technique for IoC — the container injects dependencies into objects.

### 2.2 The IoC Container and Beans

The Spring container (`ApplicationContext`) manages a set of **Beans** — a Bean is simply an object managed by the container.

Ways to declare a Bean:

```java
// 1. Component scanning: @Component and its derivatives
@Component
public class UserRepositoryImpl implements UserRepository { ... }

@Service                 // derivative of @Component, semantic
public class UserService { ... }

@Repository              // derivative of @Component, for DAO
public class UserDao { ... }

@Controller              // derivative of @Component, for controllers
public class UserController { ... }

// 2. @Bean methods: inside @Configuration classes
@Configuration
public class AppConfig {
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource(...);
    }
    @Bean
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
```

The `@Component` family — annotate the class, and Spring scans the package to create instances. `@Bean` — annotate a method, and its return value becomes a Bean; ideal for bringing third-party library classes into the container (you cannot edit their source to add `@Component`).

### 2.3 Three Styles of Dependency Injection

```java
// 1. Constructor injection (recommended!)
@Service
@RequiredArgsConstructor   // Lombok generates the constructor
public class UserService {
    private final UserRepository repo;     // final, immutable
    private final EmailService email;
}

// 2. Setter injection
@Service
public class UserService {
    private UserRepository repo;
    @Autowired
    public void setRepo(UserRepository repo) { this.repo = repo; }
}

// 3. Field injection (not recommended)
@Service
public class UserService {
    @Autowired
    private UserRepository repo;   // cannot be final, hard to test
}
```

**Why prefer constructor injection?**

- **Immutable** — `final` fields cannot be changed after injection; thread-safe.
- **Explicit dependencies** — At a glance you can see what this class needs.
- **Easy to test** — In unit tests, just `new UserService(mockRepo, mockEmail)`.
- **Avoids null pointers** — The container must supply all dependencies at construction; "forgot to inject" cannot happen.

The anti-pattern of field injection — cannot be `final`, cannot be tested outside Spring, hides circular dependencies. Spring 6+ discourages field injection.

### 2.4 Bean Scopes

| Scope | Meaning |
| --- | --- |
| `singleton` (default) | One instance per container |
| `prototype` | A new instance on every injection |
| `request` | One per HTTP request (web apps) |
| `session` | One per HTTP Session |
| `application` | One per ServletContext |

```java
@Component
@Scope("prototype")
public class Counter { ... }   // a new object on every injection
```

The vast majority of Beans are `singleton` — stateless services, DAOs, Repositories. Stateful ones (user shopping carts, session data) use `prototype`/`session`.

## 3. The Bean Lifecycle

A Bean goes through these stages from creation to destruction:

```
1. Instantiation       -- the container calls the constructor to new the object
2. Property assignment -- inject dependencies (@Autowired)
3. BeanNameAware / BeanFactoryAware and other Aware interfaces
4. BeanPostProcessor.postProcessBeforeInitialization  -- pre-initialization callback
5. @PostConstruct / InitializingBean.afterPropertiesSet / init-method  -- initialization
6. BeanPostProcessor.postProcessAfterInitialization   -- post-initialization callback (AOP proxy created here!)
7. Bean ready          -- ready for use
8. @PreDestroy / DisposableBean.destroy / destroy-method  -- destruction
```

```java
@Component
public class MyBean {
    @PostConstruct
    public void init() {
        System.out.println("Bean initialized (dependencies injected)");
    }

    @PreDestroy
    public void cleanup() {
        System.out.println("Cleaning up before Bean destruction");
    }
}
```

`@PostConstruct` runs after dependency injection is complete — commonly used to validate configuration or start background threads. `@PreDestroy` runs before the container shuts down — used to release resources.

### 3.1 AOP Proxies and the Lifecycle

AOP proxies are generated during the `BeanPostProcessor.postProcessAfterInitialization` stage — the container discovers that a Bean has annotations like `@Transactional`/`@Async` and wraps the original Bean in a proxy before injecting it. This is why `@Transactional` does not work for self-invocation — `this.method()` goes through the original object and bypasses the proxy.

## 4. Annotation-based Configuration

### 4.1 Component Scanning

```java
@Configuration
@ComponentScan(basePackages = "com.example")
public class AppConfig { ... }
```

`@ComponentScan` tells Spring to scan the specified package (and sub-packages) for all `@Component`/`@Service`/`@Repository`/`@Controller` and register them as Beans. Spring Boot's `@SpringBootApplication` includes `@ComponentScan` by default.

### 4.2 @Autowired

```java
@Service
public class UserService {
    @Autowired
    private UserRepository repo;    // inject by type

    @Autowired
    @Qualifier("userRepoImpl")      // when multiple implementations exist, select by name
    private UserRepository repo2;
}

// When multiple Beans qualify, @Primary marks the default
@Primary
@Repository
public class UserRepoImpl implements UserRepository { ... }
```

`@Autowired` injects by type by default. When multiple implementations exist, use `@Qualifier` to specify a name, or `@Primary` to mark the default implementation.

### 4.3 @Bean Methods

```java
@Configuration
public class DataSourceConfig {

    @Bean
    @ConfigurationProperties(prefix = "app.datasource")   // inject properties from config
    public DataSource dataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    public PlatformTransactionManager transactionManager(DataSource ds) {
        return new DataSourceTransactionManager(ds);   // parameter ds is auto-injected
    }
}
```

`@Bean` methods may take parameters — parameters are auto-injected from the container. This is the standard way to bring third-party library classes into the container.

## 5. AOP: Aspect-Oriented Programming

**AOP (Aspect-Oriented Programming)** — Extract "cross-cutting concerns" (transactions, logging, permissions, caching) from business code and handle them uniformly via "aspects."

### 5.1 Core AOP Concepts

| Term | Meaning | Example |
| --- | --- | --- |
| **Aspect** | Modularization of a cross-cutting concern | `LogAspect` logging aspect |
| **Join Point** | A point during program execution | A method invocation |
| **Pointcut** | Where the aspect applies (expression) | `execution(* com.example..*.*(..))` |
| **Advice** | What to do at a pointcut | `@Before`/`@After`/`@Around` |
| **Target object** | The original object being proxied | `UserService` |
| **Proxy** | The proxy object wrapping the target | CGLIB/JDK dynamic proxy |
| **Weaving** | Applying the aspect to the target | Runtime (Spring) / Compile time (AspectJ) |

### 5.2 The Five Advice Types

```java
@Aspect
@Component
public class LogAspect {

    // Pointcut: match all methods of all classes under com.example.service
    @Pointcut("execution(* com.example.service..*.*(..))")
    public void serviceMethods() {}

    // 1. Before advice
    @Before("serviceMethods()")
    public void before(JoinPoint jp) {
        System.out.println("[Before] " + jp.getSignature());
    }

    // 2. After advice (runs whether or not an exception is thrown)
    @After("serviceMethods()")
    public void after(JoinPoint jp) {
        System.out.println("[After] " + jp.getSignature());
    }

    // 3. After-returning advice (after a normal return)
    @AfterReturning(pointcut = "serviceMethods()", returning = "result")
    public void afterReturning(JoinPoint jp, Object result) {
        System.out.println("[AfterReturning] " + jp.getSignature() + " -> " + result);
    }

    // 4. After-throwing advice
    @AfterThrowing(pointcut = "serviceMethods()", throwing = "ex")
    public void afterThrowing(JoinPoint jp, Exception ex) {
        System.out.println("[AfterThrowing] " + jp.getSignature() + " throws " + ex);
    }

    // 5. Around advice (most powerful, controls whether the original method runs)
    @Around("serviceMethods()")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            Object result = pjp.proceed();    // execute the original method
            return result;
        } finally {
            long cost = System.currentTimeMillis() - start;
            System.out.println("[Around] " + pjp.getSignature() + " took " + cost + "ms");
        }
    }
}
```

`@Around` is the most powerful — it can decide whether to execute the original method, modify parameters, modify the return value, and catch exceptions. `@Transactional` is implemented with `@Around` — `try { proceed(); commit(); } catch { rollback(); }`.

### 5.3 Pointcut Expressions

```java
// execution(modifier? return-type package.class.method(param-types) exception?)
execution(public * com.example.service.*.*(..))        // public methods of all classes in the service package
execution(* com.example..*.*(..))                       // all methods in com.example and sub-packages
execution(* UserService.*(..))                          // all methods of UserService
execution(* UserService.find*(..))                      // all methods of UserService starting with find
execution(* save*(String, ..))                          // methods starting with save, first param String, rest arbitrary

// @annotation: match methods carrying a given annotation
@Pointcut("@annotation(com.example.Loggable)")
public void loggableMethods() {}

// @within: match classes carrying a given annotation
@Pointcut("@within(org.springframework.stereotype.Service)")
public void serviceClasses() {}
```

### 5.4 AOP Implementation: Dynamic Proxy

Spring AOP is built on the **dynamic proxy** we studied in the previous stage:

- If a class implements an interface → JDK dynamic proxy
- If a class implements no interface → CGLIB (generates a subclass)
- Spring Boot 2.0+ defaults to CGLIB

In Stage 10 we hand-wrote a dynamic-proxy AOP — Spring AOP's essence is simply making that mechanism standardized and powerful.

## 6. Spring Transaction Management

Transactions are a classic AOP application — `@Transactional` wraps a method in a transaction automatically.

### 6.1 Programmatic vs Declarative

```java
// Programmatic (manual management, not recommended)
@Transactional
public void transfer(Long from, Long to, BigDecimal amount) {
    TransactionStatus status = txManager.getTransaction(new DefaultTransactionDefinition());
    try {
        accountDao.debit(from, amount);
        accountDao.credit(to, amount);
        txManager.commit(status);
    } catch (Exception e) {
        txManager.rollback(status);
        throw e;
    }
}

// Declarative (recommended, @Transactional does it in one line)
@Transactional
public void transfer(Long from, Long to, BigDecimal amount) {
    accountDao.debit(from, amount);   // automatically inside a transaction
    accountDao.credit(to, amount);
}
```

`@Transactional` is essentially an AOP aspect — `@Around` wraps the method in "begin transaction → execute → commit/rollback."

### 6.2 @Transactional Configuration

```java
@Transactional(
    propagation = Propagation.REQUIRED,        // propagation behavior
    isolation = Isolation.DEFAULT,             // isolation level
    timeout = 30,                              // timeout (seconds)
    readOnly = false,                          // read-only transaction
    rollbackFor = Exception.class,             // which exceptions trigger rollback
    noRollbackFor = BusinessException.class    // which exceptions do not roll back
)
public void doSomething() { ... }
```

**Propagation** — how a transaction propagates when methods call each other:

| Propagation | Behavior |
| --- | --- |
| `REQUIRED` (default) | Join the existing transaction; create a new one if none exists |
| `REQUIRES_NEW` | Always create a new one, suspending the current transaction |
| `NESTED` | Nested transaction (savepoint) |
| `SUPPORTS` | Join if a transaction exists; otherwise run non-transactionally |
| `NOT_SUPPORTED` | Run non-transactionally, suspending the current transaction |
| `MANDATORY` | Must run inside a transaction, otherwise throw |
| `NEVER` | Must run without a transaction, otherwise throw |

**Isolation level** — how much concurrent transactions affect each other:

| Isolation | Dirty read | Non-repeatable read | Phantom read |
| --- | --- | --- | --- |
| `READ_UNCOMMITTED` | Possible | Possible | Possible |
| `READ_COMMITTED` | Not possible | Possible | Possible |
| `REPEATABLE_READ` (MySQL default) | Not possible | Not possible | Possible |
| `SERIALIZABLE` | Not possible | Not possible | Not possible |

### 6.3 Pitfalls Where @Transactional Silently Fails

1. **Self-invocation** — `this.method()` goes through the original object and bypasses the proxy. Move the method to another Bean, or inject yourself.
2. **Method is not public** — by default only public methods are proxied.
3. **Exception is swallowed** — a `try-catch` that catches and does not re-throw makes the transaction think everything is fine; no rollback.
4. **Wrong exception type** — by default only `RuntimeException` triggers rollback; checked exceptions do not. Configure `rollbackFor = Exception.class`.
5. **Class is not a Bean** — `@Transactional` on a class not managed by Spring has no effect.

## 8. Chapter Summary

| Concept | Key Points |
| --- | --- |
| IoC | Inversion of Control; object creation handed to the container |
| DI | Dependency Injection; the container injects dependencies into objects |
| Bean | A container-managed object; singleton by default |
| `@Component` family | Declares a Bean; registered via component scanning |
| `@Autowired` | Injects dependencies; constructor injection recommended |
| `@Bean` | A method's return value becomes a Bean |
| Bean lifecycle | Instantiation → injection → initialization → use → destruction |
| AOP | Aspect / pointcut / advice |
| Five advice types | Before/After/AfterReturning/AfterThrowing/Around |
| `@Transactional` | Declarative transactions, implemented via AOP |

**Memory mnemonics**:

- **IoC inverts control** — objects do not `new` themselves; the container manages them.
- **Constructor injection is safest** — final, easy to test, explicit dependencies.
- **Beans are singletons by default** — no mutable state fields.
- **AOP's five advice types** — Before before, After after, Returning on success, Throwing on exception, Around wraps all.
- **`@Transactional` is AOP** — self-invocation does not work, only public methods are proxied, exceptions must be re-thrown.
- **JDK proxy needs an interface, CGLIB uses inheritance** — Spring Boot defaults to CGLIB.

## Conclusion: From Servlet to Spring

This chapter covered Spring's two great pillars — IoC decouples object management, AOP separates cross-cutting logic.
