# Sealed Classes

This chapter covers **Sealed Classes** — a feature officially released in Java 17 (JEP 409). They are "siblings" to the Records of the previous chapter: Records solve "closed value objects," while Sealed Classes solve "closed inheritance hierarchies." Combined with Pattern Matching, Java can for the first time elegantly model a core concept of functional languages — **algebraic data types (Algebraic Data Type, ADT)**.

## 1. Why Sealed Classes: The Cost of Open Inheritance

### 1.1 Java's Traditional "Open Inheritance"

Java's class inheritance is **open** by default — any `public class` can be `extends`-ed by anyone (as long as it isn't `final`). This "openness" is good in frameworks (extensible), but often a disaster in domain modeling:

```java
// You define a Shape class
public class Shape { ... }

// You think there are only Circle/Square/Triangle subclasses
// But someone else can do this:
class WeirdShape extends Shape { ... }   // you can't control this
class MaliciousShape extends Shape { ... }
```

Problems:

- **switch doesn't dare omit default** — you don't know if `WeirdShape` will pop up in the future; `switch` must include `default`, and the compiler can't check "exhaustiveness" for you.
- **equals/hashCode are hard to write** — should `Shape.equals` consider subclasses? How do you determine "same kind"?
- **Domain constraints are broken** — the business rule "only three shapes" can't be expressed in code, only in verbal agreements.

### 1.2 What Sealed Classes Solve

**Sealed Classes** let you **explicitly declare "which classes can extend me"** — locking inheritance rights within a whitelist:

```java
public sealed class Shape permits Circle, Square, Triangle {}
```

`Shape` now only allows `Circle`, `Square`, and `Triangle` to extend it — any other class `extends Shape` is a direct compile error. This codifies "closed inheritance hierarchy" as a language feature.

### 1.3 Historical Background

Sealed Classes were officially released in Java 17 (JEP 409), after previewing in Java 15, second preview in Java 16, and finalization in Java 17. They are part of a coordinated design with Records and Pattern Matching — the three together constitute Java's "modern data modeling."

## 2. Sealed Class Definition: sealed + permits

### 2.1 Basic Syntax

```java
public sealed class Shape permits Circle, Square, Triangle {}
//                ^^^^^^                ^^^^^^^^^^^^^^^^^^^^^^^^^
//                Declared as sealed    List of permitted subclasses
```

`sealed` is a modifier; `permits` is followed by a comma-separated list of subclasses. Subclasses must be in the **same module or the same package** — this is a hard constraint of Sealed Classes.

### 2.2 The Subclass's "Pick One of Three" Modifier

Subclasses in the `permits` list **must** use one of these three modifiers:

| Modifier | Meaning | Can it be further subclassed? |
| --- | --- | --- |
| `final` | Final class, cannot be subclassed | No |
| `sealed` | Sealed class, continues `permits` its own subclasses | Yes, but must declare a whitelist |
| `non-sealed` | Non-sealed class, returns to open inheritance | Yes, anyone can extend |

```java
public sealed class Shape permits Circle, Square, Triangle {}

final class Circle extends Shape {}            // Final, cannot be subclassed
sealed class Square extends Shape permits FilledSquare, OutlinedSquare {}
non-sealed class Triangle extends Shape {}     // Open, any class can extend

// Square's subclasses
final class FilledSquare extends Square {}
final class OutlinedSquare extends Square {}

// Triangle is non-sealed, any class can extend it
class RightTriangle extends Triangle {}
class IsoscelesTriangle extends Triangle {}
```

**Why force pick one of three?** Because without forcing, subclasses default to open inheritance, and the sealed-ness is silently broken. Java chose "explicit over implicit" — if you want open, use `non-sealed`; if you want closed, use `final`; if you want half-closed, use `sealed`. You must say it explicitly.

### 2.3 Omitting permits: Same-File Subclasses

If the subclasses are in the **same source file** as the sealed class, you can omit `permits` — the compiler automatically finds direct subclasses in the same file:

```java
// Shape.java
public sealed class Shape {}   // permits omitted

final class Circle extends Shape {}
final class Square extends Shape {}
final class Triangle extends Shape {}
```

The compiler infers `Shape permits Circle, Square, Triangle`. This is a common compact style after Java 17.

### 2.4 Interfaces Can Be Sealed Too

`sealed` can modify not only classes but also **interfaces**:

```java
public sealed interface Currency permits CNY, USD, EUR {}

record CNY(long fen) implements Currency {}
record USD(long cents) implements Currency {}
record EUR(long cents) implements Currency {}
```

The combination of interface + Record + Sealed is the standard paradigm for modeling ADTs (detailed below).

## 3. Constraints on Sealed Classes

### 3.1 Same Module or Same Package

Subclasses in the `permits` list must be in the **same module** as the sealed class (if the sealed class is in a named module), or in the **same package** (if in an unnamed module). This prevents "third parties maliciously adding subclasses."

```java
// Module A
module A { exports com.example; }
public sealed class Shape permits com.example.Circle, com.other.Square {}  // Error! Square isn't in module A
```

### 3.2 Subclasses Must Directly extends

Classes in the `permits` list must **directly** extend the sealed class — not "grandchild" classes. If `Shape permits Circle`, then `Circle` must `extends Shape`; it cannot be `Circle extends Ellipse extends Shape`.

### 3.3 Subclasses Must Be Explicitly Declared

Subclasses **must** be modified with `final`/`sealed`/`non-sealed` — none can be omitted. Omitting is a compile error.

```java
sealed class Shape permits Circle {}
class Circle extends Shape {}   // Compile error! Must be final/sealed/non-sealed
```

## 4. Algebraic Data Types (ADT): The Core of Functional Languages

### 4.1 What Is an ADT

**Algebraic Data Type (ADT)** is a core feature of functional languages (Haskell, OCaml, Scala) — combining complex data structures through "product types" and "sum types."

- **Product type**: `Point(int x, int y)` — a point = x **and** y. Both present, "multiplication" combination.
- **Sum type**: `Shape = Circle | Square | Triangle` — a shape = circle **or** square **or** triangle. "Addition" combination.

Java previously had no "sum type" — any `class` could be subclassed by anyone, so you couldn't express "Shape **only has these three**." Sealed Classes finally give Java "sum types."

### 4.2 Java's Paradigm for Modeling ADTs

```java
// Sum type: Shape has only these three
public sealed interface Shape permits Circle, Rectangle, Triangle {}

// Product type: each shape has its own attributes
record Circle(double radius) implements Shape {}
record Rectangle(double width, double height) implements Shape {}
record Triangle(double a, double b, double c) implements Shape {}
```

This is the classic ADT modeling — `sealed interface` defines the "sum type," `record` defines the "product type." Each `record` is an immutable value object, and the entire inheritance hierarchy is closed.

### 4.3 The Benefit of ADTs: Exhaustiveness Checking

The biggest benefit of ADTs is **switch exhaustiveness checking** — the compiler can check "whether all cases are handled":

```java
double area(Shape shape) {
    return switch (shape) {
        case Circle(double r) -> Math.PI * r * r;
        case Rectangle(double w, double h) -> w * h;
        // case Triangle is missing! Compiler warns: not exhaustive
    };
}
```

If you later add `case Pentagon`, all switches will get compile warnings — forcing you to update every place that handles Shape. This is the "compile-time guarantee" of ADT + Pattern Matching; traditional Java can't do it.

## 5. Sealed + Record + Pattern Matching: The Combination Punch

This example shows the classic combination of all three — modeling an expression tree (AST) and destructuring it with Pattern Matching during evaluation. This is a classic scenario for compilers, rule engines, and configuration systems.

```java
// Abstract expression: only three types
sealed interface Expr permits Num, Add, Mul {}

record Num(double value) implements Expr {}
record Add(Expr left, Expr right) implements Expr {}
record Mul(Expr left, Expr right) implements Expr {}

// Evaluation
double eval(Expr expr) {
    return switch (expr) {
        case Num(double v) -> v;
        case Add(Expr l, Expr r) -> eval(l) + eval(r);
        case Mul(Expr l, Expr r) -> eval(l) * eval(r);
        // No default needed! The compiler knows there are only these three
    };
}

// Expression: (3 + 4) * 5
Expr expr = new Mul(new Add(new Num(3), new Num(4)), new Num(5));
System.out.println(eval(expr));   // 35.0
```

**`eval` needs no `default`** — `Expr` is sealed, and the compiler knows there are only `Num`/`Add`/`Mul`. This is the power of ADT + Pattern Matching — compile-time exhaustiveness guarantee; adding a `Sub` subclass in the future will immediately flag an error to remind you.

## 6. Sealed vs Final vs Abstract

| Modifier | Inheritance | When to use |
| --- | --- | --- |
| `final` | Fully closed, cannot be subclassed | Single immutable value (e.g., String) |
| `sealed` | Whitelist-closed, only specified subclasses can extend | Domain modeling "sum type," known subclasses |
| `non-sealed` | Open inheritance | "Opens a hole" inside a sealed hierarchy |
| (no modifier) | Default open inheritance | Traditional Java class |
| `abstract` | Must be subclassed to use | Template method, abstract base class |

Sealed isn't a replacement for `final`/`abstract` — it **fills the gap in between** — "whitelist-closed" between "fully open" and "fully closed."

## 8. Cooperation with Pattern Matching

The `switch` pattern matching that keeps appearing in this chapter is a Java 21 feature (detailed in the next chapter). Sealed Classes are Pattern Matching's "best partner":

- **Sealed provides "closure"** — the compiler knows all subclasses.
- **Pattern Matching provides "deconstruction"** — directly gets the components.
- **Exhaustiveness checking** — missing any one in a switch triggers a compile warning.

Without Sealed, a switch must include `default` (because unknown subclasses might exist); only with Sealed can you enjoy "omitting default + compile-time exhaustiveness."

## 9. Real Application Scenarios for Sealed

| Scenario | Example |
| --- | --- |
| Domain modeling | Payment methods, order states, user roles — business "closed enumeration" concepts |
| AST (Abstract Syntax Tree) | Expression trees, JSON/HTML nodes, rule engine rules |
| State machines | State sets of finite state machines — `sealed interface State permits Idle, Running, Paused, Done` |
| API return values | `sealed interface ApiResult permits Success, Error＜T＞` |
| Config items | `sealed interface Config permits YamlConfig, JsonConfig, EnvConfig` |

**When NOT to use Sealed**:

- Subclasses are open and third-party extensible (e.g., Spring's `BeanPostProcessor`).
- Framework extension points — deliberately open for user subclassing.
- Subclass count explodes — Sealed suits "few and stable" hierarchies.

## 10. Chapter Summary

| Concept | Key Points |
| --- | --- |
| `sealed class` | Declares a sealed class; `permits` specifies the whitelist of subclasses |
| `permits` | Lists the permitted subclasses |
| Subclass pick one of three | `final` / `sealed` / `non-sealed`, must be explicit |
| Same module/package | Subclasses must be in the same module or package as the sealed class |
| Interfaces can be sealed too | `sealed interface` |
| ADT | Algebraic Data Type = sum type (sealed) + product type (record) |
| Exhaustiveness checking | Switch pattern matching can omit default; the compiler checks exhaustiveness |
| Reflection API | `isSealed()`, `getPermittedSubclasses()` |

**Mnemonics**:

- **Sealed is "whitelist inheritance"** — only state-permits subclasses can extend.
- **`permits` is the "invitation"** — only invited subclasses can enter.
- **Subclass picks one of three** — `final` closes, `sealed` half-closes, `non-sealed` opens.
- **Same module, same package** — subclasses must "live with" the sealed class.
- **Sealed + record = ADT** — sum type + product type, functional modeling.
- **Exhaustiveness checking** — the compiler checks your switch for missing cases.

## Conclusion: Java's Functional Modeling Finally Matures

Sealed Classes are an important completion for Java in "domain modeling" — together with Records and Pattern Matching, they form modern Java data modeling's "three-piece set":

- **Records** — immutable value objects (product type).
- **Sealed Classes** — closed inheritance hierarchies (sum type).
- **Pattern Matching** — deconstruction + exhaustiveness checking (using ADTs).

These three coordinate, letting Java for the first time model algebraic data types as elegantly as Scala and Haskell.
