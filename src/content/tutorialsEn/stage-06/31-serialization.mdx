# Serialization

Objects in memory form a fragile web — objects reference objects, which reference yet more objects, all maintained by the JVM in memory. The moment the JVM shuts down, this web dissipates. If you want an object to "travel" — be saved to disk, sent to another machine, or cached in Redis — you have to "flatten" it into a byte sequence. This process is called **serialization**. Rebuilding an object from bytes is the reverse, called **deserialization**.

Java's built-in serialization mechanism (`Serializable` + `ObjectOutputStream`) is the oldest and most direct approach. It's convenient but has notorious security issues, and today it has been largely replaced by JSON/Protobuf in many scenarios. But understanding it is still necessary — it underpins Java's remote method invocation (RMI), some caches (EhCache), Spring Session, and more.

## 1. The Serializable Interface: A Marker Interface

To make a class's objects serializable, simply implement the `java.io.Serializable` interface:

```java
import java.io.Serializable;

public class Person implements Serializable {
    private String name;
    private int age;
    // ... constructor, getters
}
```

Note: `Serializable` is a **marker interface** — it **has no methods at all**. Its sole purpose is to "tag" the class — telling the JVM "objects of this class may be serialized." At runtime, `ObjectOutputStream` checks with `instanceof Serializable` and throws `NotSerializableException` if the check fails.

Why not design it as an interface with methods? Because the serialization logic is auto-generated by the JVM — it reads all fields via reflection, requiring no code from you. This "zero-intrusion" design is its elegance, and also the root of its later troubles.

## 2. ObjectOutputStream / ObjectInputStream

The core classes for serialization and deserialization:

```java
public class ObjectOutputStream extends OutputStream {
    public void writeObject(Object obj) throws IOException;   // serialize
    public void writeInt(int v);                              // write a primitive
    public void writeUTF(String s);                           // write a string
    // ...
}

public class ObjectInputStream extends InputStream {
    public Object readObject() throws IOException, ClassNotFoundException;  // deserialize
    public int readInt();
    public String readUTF();
    // ...
}
```

### 2.1 Basic Usage

```java
import java.io.*;

// Serialize
Person p = new Person("Alice", 30);
try (ObjectOutputStream out = new ObjectOutputStream(
        new FileOutputStream("person.dat"))) {
    out.writeObject(p);
}

// Deserialize
try (ObjectInputStream in = new ObjectInputStream(
        new FileInputStream("person.dat"))) {
    Person restored = (Person) in.readObject();
    System.out.println(restored);
}
```

`readObject` returns `Object`, so a cast is needed. It also throws `ClassNotFoundException` — because during deserialization the JVM must load the class named in the byte stream.

### 2.2 Serializing the Entire Object Graph

If the object you serialize references another object, the JVM **recursively serializes the entire object graph** — as long as all referenced objects implement `Serializable`:

```java
class Address implements Serializable {
    String city;
}

class Person implements Serializable {
    String name;
    Address address;   // must also be Serializable
}
```

If `Address` doesn't implement `Serializable`, serializing `Person` throws `NotSerializableException`. This rule is implicit but powerful — for an object to be serializable, every object reachable from it must be serializable too.

## 3. serialVersionUID: Version Compatibility

The serialized byte stream records the class's "fingerprint" — the `serialVersionUID` (stream version UID). During deserialization, the JVM compares **the UID in the byte stream** with **the current class's UID**:

- If they match: deserialization is allowed.
- If they don't match: `InvalidClassException` is thrown.

### 3.1 The Consequences of Not Declaring It Explicitly

If you don't explicitly declare a `serialVersionUID`, the compiler auto-computes one based on the class structure (fields, method signatures). **The problem is**: when you later add a field to the class, the auto-computed UID changes — and the previously serialized byte stream can no longer be deserialized.

```java
// v1: first release
class Person implements Serializable {
    String name;
}

// You saved 1 million person.dat files

// v2: add a field
class Person implements Serializable {
    String name;
    int age;   // after adding a field, the auto-computed UID changes
}

// All 1 million previously saved person.dat files fail to deserialize!
```

### 3.2 Declaring It Explicitly

```java
public class Person implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    private int age;
}
```

Once declared explicitly, adding a field doesn't change the UID — during deserialization, new fields retain their default values (`0`, `null`), while old fields are restored normally. This is the core purpose of the UID: **letting you control version compatibility**.

**Rule**: every `Serializable` class should explicitly declare a `serialVersionUID`. Whether you generate one with an IDE or just write `1L`, the key is "explicit."

## 4. The transient Keyword: Excluding Fields from Serialization

Some fields shouldn't be serialized — passwords, temporary caches, platform-specific resources. Mark them with `transient`:

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String username;
    private transient String password;   // will not be serialized
    private transient int loginCount = 0; // temporary statistic, not persisted
}
```

After deserialization, `transient` fields hold their type's default value (`null`, `0`, `false`).

`static` fields also don't participate in serialization — `static` belongs to the class, not the object, and serialization is a snapshot of "object state."

## 5. Custom Serialization: writeObject / readObject

The JVM's default serialization doesn't always meet your needs. For instance, if you want to encrypt a password during serialization, compress data, or skip certain logic — you can write **private `writeObject` and `readObject` methods** in the class:

```java
public class Account implements Serializable {
    private static final long serialVersionUID = 1L;
    private String username;
    private transient String password;   // transient + custom write logic

    // Private method, signature must match exactly
    private void writeObject(ObjectOutputStream out) throws IOException {
        out.defaultWriteObject();                    // first write the default fields
        out.writeUTF(encrypt(password));             // then write the encrypted password
    }

    private void readObject(ObjectInputStream in)
            throws IOException, ClassNotFoundException {
        in.defaultReadObject();                      // first read the default fields
        this.password = decrypt(in.readUTF());       // then decrypt
    }

    private String encrypt(String s) { return new StringBuilder(s).reverse().toString(); }
    private String decrypt(String s) { return new StringBuilder(s).reverse().toString(); }
}
```

During serialization, the JVM uses reflection to look for these two private methods — if they exist, it calls yours; if not, it uses the defaults. `defaultWriteObject`/`defaultReadObject` let you "add a little something on top of the default logic."

## 6. The Pitfalls of Serialization

Java serialization is convenient, but full of traps. Understanding these traps explains why modern projects avoid it.

### 6.1 Security Issues (The Most Severe)

During deserialization, the JVM loads and instantiates classes based on the class names in the byte stream — **an attacker can craft a malicious byte stream that causes the JVM to instantiate dangerous classes**, triggering arbitrary code execution. This kind of attack is called a "Java deserialization vulnerability." The 2015 Apache Commons Collections vulnerability shocked the industry, affecting countless systems including WebLogic, Jenkins, and WebSphere.

**Core principle**: **never deserialize untrusted data.** `readObject` is an "opening Pandora's box" operation — it can invoke the construction logic of arbitrary classes.

### 6.2 Version Compatibility

Adding fields, changing field types, moving a class to a different package — all can cause old data to fail deserialization or get fields mixed up. `serialVersionUID` is only a "soft contract" and can't solve every compatibility problem.

### 6.3 Performance and Size

Java serialization's byte stream contains a lot of class information (class name, field names, UID), making it **bulky** — serializing a simple object can take hundreds of bytes. It's also several times slower than JSON libraries.

### 6.4 Not Cross-Language

Java serialization is a Java-proprietary format — Python, Go, and Node.js can't read it. This is a fatal flaw in polyglot microservice environments.

## 7. Alternatives: JSON and Protobuf

In new projects, serialization is mostly done with JSON or Protobuf, not Java's built-in serialization.

### 7.1 JSON

JSON is a text format — readable, cross-language, and debug-friendly. There are two mainstream Java libraries:

- **Jackson**: Spring's default integration, feature-rich, excellent performance.
- **Gson**: Made by Google, with a clean API, suited for simple scenarios.

```java
// Jackson example (pseudo-code, requires com.fasterxml.jackson.databind.ObjectMapper)
ObjectMapper mapper = new ObjectMapper();
String json = mapper.writeValueAsString(person);          // object -> JSON
Person p = mapper.readValue(json, Person.class);          // JSON -> object
```

JSON's advantages: readable, cross-language, good compatibility with field additions/removals (missing fields use defaults, extra fields are ignored). Disadvantages: larger than binary formats, no native binary data support (you need Base64).

### 7.2 Protobuf

Google's Protocol Buffers is a binary format — **compact, fast, strongly typed**. You write a `.proto` file to define the structure, and the compiler generates code for each language.

```proto
// person.proto
syntax = "proto3";
message Person {
    string name = 1;
    int32 age = 2;
}
```

Protobuf is suited for high-performance scenarios — RPC communication (gRPC uses Protobuf by default), massive data storage. Disadvantages: not human-readable (binary), requires a schema file.

### 7.3 Choosing the Right Tool

| Scenario | Recommendation |
| --- | --- |
| Config files, API responses | JSON |
| Internal caches, sessions | JSON or Java serialization (if trusted) |
| High-performance RPC | Protobuf |
| Cross-language big data | Protobuf / Avro |
| Java-proprietary legacy systems | Java serialization (not recommended for new projects) |

## 9. Quick Reference Table

| Concept | Description |
| --- | --- |
| `Serializable` | Marker interface, no methods, indicates an object is serializable |
| `ObjectOutputStream.writeObject` | Serializes an object to a stream |
| `ObjectInputStream.readObject` | Deserializes an object from a stream |
| `serialVersionUID` | Class version fingerprint; declare explicitly to control compatibility |
| `transient` | Field is excluded from serialization |
| `static` | Belongs to the class, not the object; not serialized |
| `writeObject`/`readObject` | Private methods for custom serialization logic |
| `defaultWriteObject`/`defaultReadObject` | Invoke default serialization within custom logic |

## Conclusion: Convenient, but at a Cost

Java serialization is a feature with historical baggage. Its design — "marker interface + JVM auto-reflection of fields" — is elegant and zero-intrusion, making serialization nearly transparent to business code. But that very transparency is where its problems lie:

- **Security**: the auto-deserialization mechanism lets attackers trigger arbitrary class code, causing RCE.
- **Compatibility**: change the class and old data may break; `serialVersionUID` is only a stopgap.
- **Cross-language**: a purely Java-proprietary format, unusable in polyglot environments.
- **Performance**: bulky and slow, outperformed by JSON/Protobuf.

In new projects, **JSON** is the first choice for cross-language data exchange (readable, debug-friendly), and **Protobuf** for high-performance scenarios (compact, fast). Java's built-in serialization survives in two corners: legacy system compatibility, and private inter-process communication in Java (RMI, JGroups, etc.) — scenarios where "both sides are Java and the data is trusted," and the convenience of serialization shines.

Understanding serialization isn't just about learning an API — it's about understanding the fundamental question of "how object state travels through time and space." Whether it's Java serialization, JSON, or Protobuf, they're all answering the same question: **how to losslessly flatten and restore that web of objects in memory.**
