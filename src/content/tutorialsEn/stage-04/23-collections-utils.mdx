# The Collections Utility Class

If the earlier `List`, `Set`, and `Map` are various "containers," then `Collections` is the "Swiss Army knife" for this set of containers—it provides a bunch of static utility methods that make containers easier to use: sorting, reversing, shuffling, searching, immutable wrappers, synchronized wrappers... In addition, the `List.of`/`Set.of`/`Map.of` factory methods introduced in Java 9 and `copyOf` in Java 10 make "creating an immutable collection" easier than ever.

In this chapter, we take these tools out one by one and see what each is good for.

## 1. Collections.sort: Sorting

`Collections.sort(list)` is the classic sorting method. It requires elements to implement `Comparable`, or you can pass a `Comparator`:

```java
List<Integer> nums = new ArrayList<>(List.of(5, 1, 3, 2, 4));
Collections.sort(nums);                 // [1, 2, 3, 4, 5]
Collections.sort(nums, Comparator.reverseOrder());   // [5, 4, 3, 2, 1]
```

Under the hood: before Java 7 it used merge sort; from Java 7 onward it switched to **TimSort** (a hybrid of merge and insertion sort, particularly fast on "partially ordered" data). `List.sort()` actually converts the list to an array, sorts it, then writes it back—so `ArrayList` sorts quickly, while `LinkedList` sorts slowly (it has to convert to an array first).

> 💡 **From Java 8 onward**, `list.sort(comparator)` is preferred—it's a default method on the `List` interface and is more "object-oriented" than `Collections.sort`.

## 2. reverse and shuffle

### 2.1 reverse: Reversing

```java
List<Integer> list = new ArrayList<>(List.of(1, 2, 3, 4, 5));
Collections.reverse(list);    // [5, 4, 3, 2, 1]
```

It's implemented by "swapping head and tail," O(n).

### 2.2 shuffle: Shuffling

```java
Collections.shuffle(list);              // randomly shuffle
Collections.shuffle(list, new Random(42));   // specify a random seed (reproducible)
```

The shuffling algorithm is the classic **Fisher-Yates**—traverse from back to front, swapping each element with a random position before it. This guarantees every permutation appears with equal probability.

`shuffle` is commonly used in game development (card shuffling), machine learning (data shuffling), and random sampling.

## 3. swap and fill

```java
List<Integer> list = new ArrayList<>(List.of(1, 2, 3, 4, 5));
Collections.swap(list, 0, 4);    // swap positions 0 and 4 → [5, 2, 3, 4, 1]

Collections.fill(list, 0);       // fill everything with 0 → [0, 0, 0, 0, 0]
```

`swap` exchanges two positions; `fill` replaces all elements with a specified value. Simple but practical.

## 4. Other Common Methods

```java
List<Integer> list = new ArrayList<>(List.of(1, 2, 3, 4, 5));

Collections.max(list);                      // 5
Collections.min(list);                      // 1
Collections.frequency(list, 3);             // 1 (number of occurrences)
Collections.indexOfSubList(list, List.of(3,4));   // 2
Collections.replaceAll(list, 3, 30);        // replace all 3s with 30
Collections.rotate(list, 2);                // rotate right by 2
Collections.disjoint(List.of(1,2), List.of(3,4)); // true (no intersection)
```

`rotate(list, distance)` is interesting—it "rotates" the list right by `distance` positions:

```java
[1, 2, 3, 4, 5] rotate 2 → [4, 5, 1, 2, 3]
```

## 5. Immutable Wrappers: unmodifiableXxx

Sometimes you want to make a collection "read-only"—give it to others but don't let them modify it. `Collections.unmodifiableXxx` wraps a collection in a "read-only shell"; calling any modification method throws `UnsupportedOperationException`:

```java
List<String> mutable = new ArrayList<>(List.of("A", "B", "C"));
List<String> immutable = Collections.unmodifiableList(mutable);

immutable.add("D");   // ❌ UnsupportedOperationException
immutable.set(0, "X"); // ❌
immutable.get(0);      // ✅ you can read
```

**Note**: `unmodifiableList` is a **view**—it shares data with the original collection. Modifying the original collection changes the read-only view too:

```java
mutable.add("D");
System.out.println(immutable);   // [A, B, C, D]!
```

So "immutable" here means "cannot be modified through this reference," not "never changes." For a true snapshot that's immutable, use `List.copyOf` (see below).

Corresponding methods:

- `Collections.unmodifiableList(list)`
- `Collections.unmodifiableSet(set)`
- `Collections.unmodifiableMap(map)`
- `Collections.unmodifiableCollection(c)`

## 6. Synchronized Wrappers: synchronizedXxx

Regular `ArrayList` and `HashMap` are not thread-safe. `Collections.synchronizedXxx` wraps them in a synchronized shell, adding `synchronized` to every method:

```java
List<String> syncList = Collections.synchronizedList(new ArrayList<>());
Map<String, Integer> syncMap = Collections.synchronizedMap(new HashMap<>());
Set<String> syncSet = Collections.synchronizedSet(new HashSet<>());
```

Thread-safe, but **poor performance**—every operation locks the entire collection, resulting in very low concurrency. Modern Java prefers dedicated concurrent containers like `CopyOnWriteArrayList`, `ConcurrentHashMap`, and `ConcurrentSkipListSet`.

**Key pitfall**: you still need to synchronize manually when iterating!

```java
List<String> syncList = Collections.synchronizedList(new ArrayList<>());
// ...
synchronized (syncList) {     // must synchronize manually!
    for (String s : syncList) {
        System.out.println(s);
    }
}
```

Because the iterator itself is not synchronized—if another thread modifies the list during iteration, it still throws `ConcurrentModificationException`. This is the most common pitfall of `synchronizedXxx`.

## 7. Java 9+: Immutable Factory Methods

Java 9 introduced a set of **immutable collection factory methods**—one line of code creates a truly immutable collection:

### 7.1 List.of / Set.of / Map.of

```java
List<String> list = List.of("A", "B", "C");
Set<Integer> set = Set.of(1, 2, 3);
Map<String, Integer> map = Map.of("a", 1, "b", 2);   // up to 10 pairs
Map<String, Integer> bigMap = Map.ofEntries(
    Map.entry("a", 1),
    Map.entry("b", 2),
    Map.entry("c", 3)
);
```

Characteristics:

1. **Immutable**: `add`/`remove`/`set` all throw `UnsupportedOperationException`.
2. **No null allowed**: passing null throws `NullPointerException`.
3. **Set/Map don't allow duplicates**: `Set.of(1, 1)` throws `IllegalArgumentException`.
4. **Serialization-friendly**: the underlying implementation is optimized and uses less memory than `unmodifiableXxx`.

### 7.2 Differences from Arrays.asList / unmodifiableList

| Feature | `Arrays.asList` | `unmodifiableList` | `List.of` |
| --- | --- | --- | --- |
| Mutability | Can modify elements, but not add/remove | Fully immutable | Fully immutable |
| null | Allowed | Allowed | **Not allowed** |
| Is a view | Yes (shares original array) | Yes (shares original collection) | No (independent) |

`List.of` is "truly immutable"—it doesn't depend on any original collection; it's an independent immutable object.

### 7.3 Java 10+: copyOf

`List.copyOf`, `Set.copyOf`, and `Map.copyOf` create immutable copies from an existing collection:

```java
List<String> mutable = new ArrayList<>(List.of("A", "B", "C"));
List<String> snapshot = List.copyOf(mutable);   // immutable snapshot

mutable.add("D");
System.out.println(snapshot);   // [A, B, C] (unaffected by the original)
```

`copyOf` is the elegant way to write "defensive copies"—previously you'd need `Collections.unmodifiableList(new ArrayList＜＞(original))`, now one line of `List.copyOf(original)` does it.

Note: if the input is already an immutable collection created by `List.of`, `copyOf` won't copy—it returns the original object directly. This is an optimization.

## 8. Hands-On: Handling a Deck of Cards with Collections

<CodeRunner code={`import java.util.*;

public class Main {

## 9. Collections vs Collection

Don't confuse these two nearly identically named things:

- `Collection＜E＞`—an **interface**, the root of all single-element containers (the parent interface of List/Set/Queue).
- `Collections`—a **utility class**, full of static methods, providing algorithms like sorting, searching, and wrapping.

One is "the container itself"; the other is "the toolbox for operating on containers."

## Conclusion: Tools Make Containers Stronger

The `Collections` utility class and factory methods like `List.of` are "amplifiers" for the collections framework—they don't create new containers, but they make existing ones easier to use and safer.

Remember a few principles:

1. **When returning a collection to the outside, use `copyOf` or `unmodifiableXxx`**—protect internal state from modification.
2. **For constant collections, use `List.of` / `Set.of` / `Map.of`**—far more concise than `new ArrayList＜＞` + `add`.
3. **For concurrency, use dedicated concurrent containers, not `synchronizedXxx`**—`ConcurrentHashMap` and friends are the modern answer.
4. **`Collections.sort` is history**—from Java 8+, `list.sort()` is more idiomatic.
