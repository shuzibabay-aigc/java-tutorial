# Map

If `List` and `Set` are containers for "a bunch of things," then `Map` is a container for "a bunch of correspondences"—one key maps to one value, like a dictionary, an address book, or a configuration table. Look up a number by name, look up a translation by word, look up a user by id—these are all `Map`'s home turf.

`Map` is the most storied branch of the collections framework. Its star member `HashMap` went through the major "linked-list-to-red-black-tree" upgrade in Java 8; `LinkedHashMap` can build an LRU cache with a single configuration line; `ConcurrentHashMap` is the backbone of concurrent programming. In this chapter, we tell each of these stories in detail.

## 1. The Map Interface

`Map<K, V>` is a standalone interface that does not extend `Collection`:

```java
public interface Map<K, V> {
    // Basic operations
    V put(K key, V value);              // store (updates if key exists, returns old value)
    V get(Object key);                  // retrieve
    V getOrDefault(Object key, V defaultValue);   // retrieve, or return default if absent
    V remove(Object key);               // delete
    boolean containsKey(Object key);
    boolean containsValue(Object value);

    // Bulk
    void putAll(Map<? extends K, ? extends V> m);

    // Views
    Set<K> keySet();                         // Set view of keys
    Collection<V> values();                  // Collection view of values
    Set<Map.Entry<K, V>> entrySet();         // Set view of key-value pairs

    // Java 8+
    V putIfAbsent(K key, V value);
    V compute(K key, BiFunction<K, V, V> remappingFunction);
    V merge(K key, V value, BiFunction<V, V, V> remappingFunction);
    void forEach(BiConsumer<K, V> action);
}
```

The three view methods are the bridge between `Map` and the `Collection` world—`keySet()`, `values()`, and `entrySet()` all return **views**, and modifications to a view are reflected in the Map itself.

## 2. HashMap: Array + Linked List + Red-Black Tree

### 2.1 Core Structure

`HashMap` is the "default spokesperson" for `Map`. To understand it, focus on three fields:

```java
public class HashMap<K,V> extends AbstractMap<K,V>
        implements Map<K,V>, Cloneable, Serializable {
    transient Node<K,V>[] table;      // hash bucket array (buckets)
    transient int size;                // number of elements
    int threshold;                     // resize threshold = capacity * loadFactor
    final float loadFactor;            // load factor

    static class Node<K,V> implements Map.Entry<K,V> {
        final int hash;
        final K key;
        V value;
        Node<K,V> next;                // next node in the same bucket (linked list)
    }
}
```

`table` is a `Node[]` array, where each slot is called a "bucket." A key's hash determines which bucket it goes into, and elements in the same bucket are linked together in a linked list.

### 2.2 The Complete put Flow

`HashMap.put` does roughly the following:

1. **Compute hash**: `(h = key.hashCode()) ^ (h ＞＞＞ 16)`—XOR the high and low bits to make the hash more uniform.
2. **Determine bucket**: `hash & (capacity - 1)`—bitwise AND instead of modulo, which requires the capacity to be a power of 2.
3. **Bucket empty**: place a new `Node` directly.
4. **Bucket non-empty**: traverse the linked list/red-black tree; if the same key is found, update the value; otherwise, append.
5. **Linked list length >= 8 and capacity >= 64**: convert the linked list to a red-black tree (treeify).
6. **size > threshold**: resize.

```java
// Simplified putVal (Java 8+)
final V putVal(int hash, K key, V value, boolean onlyIfAbsent, boolean evict) {
    Node<K,V>[] tab = table;
    int n = tab.length;
    int i = (n - 1) & hash;                 // determine bucket
    Node<K,V> p = tab[i];
    if (p == null) {
        tab[i] = newNode(hash, key, value, null);   // bucket empty, place directly
    } else {
        // bucket non-empty, traverse to find/append
        // ...
    }
    if (++size > threshold) resize();       // resize
    return null;
}
```

### 2.3 Why Capacity Must Be a Power of 2

`HashMap` uses `hash & (capacity - 1)` instead of `hash % capacity` to determine the bucket—bitwise AND is much faster than modulo. But `& (capacity - 1)` is equivalent to `% capacity` **only when capacity is a power of 2**. That's why `HashMap` always rounds the capacity up to a power of 2 internally (default 16, doubled on resize).

### 2.4 Linked-List-to-Red-Black-Tree (Java 8+)

In Java 7's HashMap, severe hash collisions resulted in long linked lists in a bucket, making lookups O(n). Maliciously crafted keys could even cause a DoS attack.

Java 8 introduced the "list treeification" mechanism:

- When a linked list reaches length **8** and the `table` capacity reaches **64**: the list is converted to a red-black tree, reducing lookups to O(log n).
- When the number of tree nodes drops below **6**: it degrades back to a linked list (untreeify).

The "hysteresis" between 8 and 6 avoids frequent switching back and forth between list and tree.

### 2.5 Resizing and Rehashing

When `size ＞ threshold` (default `capacity * 0.75`), `HashMap` resizes:

- **Capacity doubles**: `newCapacity = oldCapacity ＜＜ 1`.
- **Redistribution**: each element is relocated to its new bucket. Because the capacity doubles, the new bucket index is either `original index` or `original index + original capacity`—this clever property means rehashing doesn't require recomputing the hash.

Resizing is O(n)—all elements must be redistributed. That's why **estimating capacity** is important:

```java
// Knowing you'll store 1000 elements, avoid resizing
int cap = (int) (1000 / 0.75) + 1;   // 1334, rounded up to a power of 2 = 2048
Map<String, Integer> map = new HashMap<>(2048);
```

> 💡 **Guava's Maps.newHashMapWithExpectedSize(expectedSize)** does this calculation for you—no need to compute by hand.

### 2.6 The Meaning of the 0.75 Load Factor

The load factor controls "how full before resizing":

- Default `0.75`: resize when 75% full.
- **Smaller (e.g., 0.5)**: fewer collisions, faster lookups, but more wasted space and more frequent resizes.
- **Larger (e.g., 1.0)**: better space utilization, but more collisions and slower lookups.

`0.75` is a trade-off between time and space—this is the optimal value tuned by experts like Joshua Bloch; don't change it in everyday use.

### 2.7 Crucial: Keys Must Be "Immutable"

`HashMap` uses a key's hash to locate its bucket. If the key is a mutable object and is modified after being put in, the hash changes—and `get` will no longer find it!

```java
List<String> key = new ArrayList<>(List.of("A"));
Map<List<String>, Integer> map = new HashMap<>();
map.put(key, 1);
key.add("B");                  // modified the key!
map.get(key);                  // null! hash changed, can't find the original bucket
map.get(new ArrayList<>(List.of("A")));   // also can't find it
```

**Best practice**: Use immutable objects like `String`, `Integer`, and `Long` as keys. When using custom objects as keys, either make them immutable or ensure the fields used as keys are never modified.

## 3. TreeMap: Sorted by Key

`TreeMap` stores entries in a red-black tree, so iteration yields keys in ascending order:

```java
TreeMap<String, Integer> map = new TreeMap<>();
map.put("banana", 2);
map.put("apple", 5);
map.put("cherry", 3);
map.forEach((k, v) -> System.out.println(k + "=" + v));
// apple=5, banana=2, cherry=3 (by key lexicographic order)
```

The sorting method is the same as TreeSet: natural ordering (key implements Comparable) or custom ordering (pass a Comparator to the constructor). It implements `NavigableMap`, providing navigation methods like `firstKey`/`lastKey`/`ceilingKey`/`floorKey`/`subMap`.

Suited for scenarios that need "iterate in sorted key order" or "range queries."

## 4. LinkedHashMap: Order Preservation and accessOrder

### 4.1 Two Ordering Modes

`LinkedHashMap` adds a doubly-linked list on top of `HashMap` to record order. It has two modes:

- **Insertion order** (default): ordered by the sequence of `put` calls. Iteration follows insertion order.
- **Access order** (accessOrder=true): every `get`/`put` of an existing key moves that entry to the end of the linked list—"most recently accessed goes to the back."

```java
// Insertion order (default)
LinkedHashMap<String, Integer> m1 = new LinkedHashMap<>(16, 0.75f, false);
// Access order
LinkedHashMap<String, Integer> m2 = new LinkedHashMap<>(16, 0.75f, true);
```

### 4.2 The Principle Behind LRU Cache

In `accessOrder=true` mode, the head of the linked list is "least recently accessed," and the tail is "most recently accessed"—this is exactly the basis for the **LRU** (Least Recently Used) cache eviction strategy.

Even better, `LinkedHashMap` provides a hook:

```java
protected boolean removeEldestEntry(Map.Entry<K,V> eldest) {
    return false;   // don't remove by default
}
```

This method is called after every `put`; returning `true` automatically removes the oldest entry. **Override it to return `size() ＞ capacity`, and an LRU cache is complete**—which is why "LRU cache" is a classic interview question.

## 5. ConcurrentHashMap: Thread-Safe

### 5.1 Why Not Hashtable or Collections.synchronizedMap

`Hashtable` adds `synchronized` to every method—even concurrent reads are locked, hurting performance; it has been left behind by the times. `Collections.synchronizedMap` follows the same "lock everything" approach.

`ConcurrentHashMap`'s approach is to **reduce lock granularity**—it only locks "the bucket being written to," while reads and writes to other buckets are not blocked.

### 5.2 Java 8 Implementation: CAS + synchronized Locking the Bucket

Java 7's `ConcurrentHashMap` used "segmented locks" (Segment), splitting the entire map into 16 segments, each with its own lock. Java 8 rewrote it with finer-grained "CAS + synchronized locking the head node of a single bucket":

- **Bucket empty**: use CAS to place the new node—lock-free.
- **Bucket non-empty**: `synchronized` locks the bucket's head node, appending to the linked list/tree.
- **size**: uses the `LongAdder` approach—multiple `baseCount` + `CounterCell` arrays—to reduce contention.

This design makes reads completely lock-free, and writes only lock a single bucket—delivering extremely high concurrency.

### 5.3 Atomic Operations: putIfAbsent / compute / merge

In concurrent scenarios, "check-then-act" is not atomic:

```java
// ❌ not thread-safe
if (!map.containsKey(k)) {
    map.put(k, v);
}

// ✅ atomic
map.putIfAbsent(k, v);
```

`ConcurrentHashMap` provides a set of atomic methods:

- `putIfAbsent(key, value)`: put only if absent.
- `compute(key, (k, v) -＞ ...)`: atomically compute a new value from the current one.
- `merge(key, value, (oldV, newV) -＞ ...)`: atomically merge.

```java
// Thread-safe word frequency counting
ConcurrentHashMap<String, Long> freq = new ConcurrentHashMap<>();
freq.merge(word, 1L, Long::sum);   // atomic: put 1 if absent, otherwise accumulate
```

1. Pass `accessOrder=true` to the constructor—enabling access-order mode.

## 7. HashMap Tuning Guide

| Parameter | Default | Tuning Advice |
| --- | --- | --- |
| Initial capacity | 16 | If you know the element count → set to `expected / 0.75` (rounded to a power of 2) |
| Load factor | 0.75 | Usually don't touch; if memory is tight, try 0.9; if lookups are frequent, try 0.5 |
| Key type | - | Prefer immutable objects (String, Integer) |

A common pitfall: `new HashMap＜＞(expectedSize)` can still resize. Because threshold = capacity × loadFactor, when capacity is 100, threshold is only 75. If you expect to store 100 entries, use `new HashMap＜＞(expectedSize / 0.75f + 1)`, or use Guava's `Maps.newHashMapWithExpectedSize`.

## 8. Comparison of the Four Maps

| Feature | HashMap | LinkedHashMap | TreeMap | ConcurrentHashMap |
| --- | --- | --- | --- | --- |
| Order | None | Insertion/access order | Sorted by key | None |
| null key | Allows 1 | Allows 1 | Not allowed | Not allowed |
| null value | Allows multiple | Allows multiple | Allows multiple | Not allowed |
| Thread-safe | No | No | No | Yes |
| add/get | O(1) | O(1) | O(log n) | O(1) |
| Best for | General use | Order preservation/caching | Sorting/range | Concurrency |

## Conclusion: Map Is the Crown of the Collections Framework

`Map` is more complex than `List`/`Set`, and also more powerful. It takes "key-value mapping" to the extreme: `HashMap` trades a hash table for O(1) peak performance, `TreeMap` trades a red-black tree for sorting capability, `LinkedHashMap` trades a linked list for order and LRU, and `ConcurrentHashMap` trades fine-grained locking for high concurrency.

Once you understand `HashMap`'s array + linked list + red-black tree, you understand `HashSet`; once you understand `LinkedHashMap`'s accessOrder, you hold the key to LRU caches; once you understand `ConcurrentHashMap`'s bucket locking, you've touched the gateway to concurrent programming.
