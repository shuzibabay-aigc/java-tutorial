# Build Tools: Maven / Gradle

Welcome to Stage 12 — **Engineering and Ecosystem**. In the previous eleven stages every piece of code we wrote was essentially "single-file": one `Main.java`, compiled with `javac`, run with `java`. Real projects don't look like that — dozens of modules, hundreds of dependencies, automated tests, packaging and releases. Hand-typing `javac` simply won't cut it. This stage looks at how to turn "code" into an "engineering project". The first step is the **build tool** — Maven and Gradle.

The core responsibilities of a build tool:

1. **Dependency management** — automatically download jars and their transitive dependencies from remote repositories.
2. **Build lifecycle** — compile → test → package → install → deploy, all in one command.
3. **Standardized project structure** — convention over configuration, everyone shares the same directory layout.
4. **Multi-module support** — split large projects into modules and build them together.

## 1. Maven

**Maven** (from Yiddish, "accumulator of knowledge") is the build tool open-sourced by Apache in 2002 and still the de facto standard in the Java industry. Its core philosophy is **Convention over Configuration** — you don't have to tell it where the source code is or where compiled output goes; just follow its conventions.

### 1.1 Standard Directory Structure

```
my-app/
├── pom.xml                 # Project Object Model
├── src/
│   ├── main/
│   │   ├── java/           # Main source code
│   │   └── resources/      # Resource files (config, properties)
│   └── test/
│       ├── java/           # Test source code
│       └── resources/      # Test resources
└── target/                 # Build output (auto-generated by Maven)
    ├── classes/
    └── my-app-1.0.jar
```

As long as you place code following this structure, Maven knows how to compile, test, and package it. Not a single line of configuration required.

### 1.2 POM and Coordinates

`pom.xml` is the heart of Maven — it describes project information, dependencies, and build configuration. Each project has a "coordinate" that uniquely identifies it:

```xml
<project xmlns="http://maven.apache.org/POM/4.0.0">
    <modelVersion>4.0.0</modelVersion>

    <!-- Three coordinate elements: groupId / artifactId / version -->
    <groupId>com.example</groupId>          <!-- Organization/company (reverse domain) -->
    <artifactId>my-app</artifactId>          <!-- Project name -->
    <version>1.0.0</version>                 <!-- Version number -->
    <packaging>jar</packaging>               <!-- Package type: jar/war/pom -->

    <properties>
        <maven.compiler.source>17</maven.compiler.source>
        <maven.compiler.target>17</maven.compiler.target>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
    </properties>

    <dependencies>
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <version>5.10.0</version>
            <scope>test</scope>
        </dependency>
        <dependency>
            <groupId>com.google.code.gson</groupId>
            <artifactId>gson</artifactId>
            <version>2.10.1</version>
        </dependency>
    </dependencies>
</project>
```

The **three coordinate elements** `groupId:artifactId:version` uniquely identify a jar — for example `com.google.code.gson:gson:2.10.1`. Maven repositories organize their directories exactly by these coordinates.

### 1.3 Dependency Scope

`＜scope＞` determines in which phases a dependency is available:

| Scope | Compile | Test | Runtime | Package | Example |
| --- | --- | --- | --- | --- | --- |
| `compile` (default) | ✅ | ✅ | ✅ | ✅ | Spring, Gson |
| `test` | ❌ | ✅ | ❌ | ❌ | JUnit, Mockito |
| `provided` | ✅ | ✅ | ❌ | ❌ | Servlet API (provided by container) |
| `runtime` | ❌ | ✅ | ✅ | ✅ | JDBC driver |
| `system` | ✅ | ✅ | ❌ | ❌ | Local jar (not recommended) |
| `import` | - | - | - | - | Import a BOM |

```xml
<!-- JDBC driver: only needed at runtime, not at compile time -->
<dependency>
    <groupId>mysql</groupId>
    <artifactId>mysql-connector-java</artifactId>
    <version>8.0.33</version>
    <scope>runtime</scope>
</dependency>

<!-- Servlet API: provided by Tomcat, don't include it in the package -->
<dependency>
    <groupId>jakarta.servlet</groupId>
    <artifactId>jakarta.servlet-api</artifactId>
    <version>6.0.0</version>
    <scope>provided</scope>
</dependency>
```

### 1.4 Transitive Dependencies and Conflicts

Maven handles **transitive dependencies** automatically — you depend on A, A depends on B, and Maven pulls in B for you. But this can trigger **version conflicts**:

```
project -> A (depends on B:1.0)
project -> C (depends on B:2.0)
```

Should B be 1.0 or 2.0? Maven's strategy is **nearest definition wins** — direct dependencies beat transitive ones. At the same depth, **the first declared one wins**. You can use `＜exclusions＞` to exclude a particular transitive dependency:

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>A</artifactId>
    <version>1.0</version>
    <exclusions>
        <exclusion>
            <groupId>com.example</groupId>
            <artifactId>B</artifactId>   <!-- Exclude B brought in by A -->
        </exclusion>
    </exclusions>
</dependency>
```

To investigate conflicts, run `mvn dependency:tree` — it prints the full dependency tree and makes everything clear at a glance.

### 1.5 Lifecycle and Commands

Maven has three independent lifecycles:

- **clean** — `pre-clean` → `clean` → `post-clean` (cleans the target directory)
- **default** — `validate` → `compile` → `test` → `package` → `verify` → `install` → `deploy`
- **site** — generates a project documentation site

Each phase binds to zero or more plugin goals. Common commands:

```bash
mvn clean              # Clean target
mvn compile            # Compile main code
mvn test               # Run tests
mvn package            # Package (jar/war)
mvn install            # Package and install to local repository ~/.m2/repository
mvn deploy             # Deploy to remote repository
mvn clean package      # Clean + package (common combo)
mvn dependency:tree    # View dependency tree
mvn -DskipTests package  # Package but skip tests
```

After `mvn install`, other projects can depend on `com.example:my-app:1.0.0`. `mvn deploy` publishes to a private repository such as Nexus or Artifactory.

### 1.6 Repositories and Mirrors

Maven repositories come in three tiers:

1. **Local repository** `~/.m2/repository/` — cached after the first download.
2. **Private repository** (Nexus/Artifactory) — shared within a company.
3. **Central repository** `https://repo.maven.apache.org/maven2/` — the global public one.

Lookup order: local → private → central.

Accessing the central repository from inside China is slow, so configure a mirror (`~/.m2/settings.xml`) to go through Aliyun:

```xml
<settings>
    <mirrors>
        <mirror>
            <id>aliyun</id>
            <mirrorOf>central</mirrorOf>
            <name>Aliyun Maven</name>
            <url>https://maven.aliyun.com/repository/public</url>
        </mirror>
    </mirrors>
</settings>
```

### 1.7 Multi-Module Projects

Large projects are split into multiple modules, with a parent POM managing them uniformly:

```
my-project/
├── pom.xml                  # Parent POM (packaging=pom)
├── my-domain/               # Domain model
│   └── pom.xml
├── my-service/              # Business logic
│   └── pom.xml
├── my-web/                  # Web interface
│   └── pom.xml
└── my-app/                  # Bootstrap module
    └── pom.xml
```

Parent POM:

```xml
<project>
    <groupId>com.example</groupId>
    <artifactId>my-project</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>

    <modules>
        <module>my-domain</module>
        <module>my-service</module>
        <module>my-web</module>
        <module>my-app</module>
    </modules>

    <!-- Centrally manage dependency versions (declares only, does not introduce) -->
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-dependencies</artifactId>
                <version>3.2.0</version>
                <type>pom</type>
                <scope>import</scope>
            </dependency>
        </dependencies>
    </dependencyManagement>
</project>
```

Child modules inherit the parent POM, saving repetitive configuration:

```xml
<project>
    <parent>
        <groupId>com.example</groupId>
        <artifactId>my-project</artifactId>
        <version>1.0.0</version>
    </parent>
    <artifactId>my-service</artifactId>

    <dependencies>
        <!-- Depend on another module -->
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>my-domain</artifactId>
            <version>${project.version}</version>
        </dependency>
        <!-- Spring Boot dependencies need no version (managed by parent POM) -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
        </dependency>
    </dependencies>
</project>
```

`dependencyManagement` only declares versions without actually pulling dependencies — child modules omit `＜version＞` when declaring `＜dependency＞`. This is the key mechanism for keeping version numbers consistent across modules.

## 2. Gradle

**Gradle** is the next-generation build tool open-sourced in 2012. It combines Maven's dependency management with Ant's flexibility, replacing XML with a **Groovy/Kotlin DSL** — more concise and fully programmable.

### 2.1 build.gradle

```groovy
plugins {
    id 'java'
    id 'org.springframework.boot' version '3.2.0'
    id 'io.spring.dependency-management' version '1.1.4'
}

group = 'com.example'
version = '1.0.0'

java {
    sourceCompatibility = JavaVersion.VERSION_17
    targetCompatibility = JavaVersion.VERSION_17
}

repositories {
    mavenCentral()                       // Central repository
    maven { url 'https://maven.aliyun.com/repository/public' }
}

dependencies {
    // Scopes are declared with keywords, more concise than Maven's <scope>
    implementation 'com.google.code.gson:gson:2.10.1'        // compile + runtime
    testImplementation 'org.junit.jupiter:junit-jupiter:5.10.0'   // test only
    runtimeOnly 'mysql:mysql-connector-java:8.0.33'          // runtime only
    compileOnly 'jakarta.servlet:jakarta.servlet-api:6.0.0'  // compile only

    // Under Spring Boot dependency management, no version needed
    implementation 'org.springframework.boot:spring-boot-starter-web'
}

test {
    useJUnitPlatform()
}
```

Gradle's dependency scope keywords:

- `implementation` — compile + runtime (similar to Maven's `compile`, but not exposed to upstream consumers).
- `api` — compile + runtime + exposed to upstream (used by library projects).
- `compileOnly` — compile only (similar to `provided`).
- `runtimeOnly` — runtime only (similar to `runtime`).
- `testImplementation` — test compile and runtime only (`test`).

### 2.2 Tasks and Lifecycle

The core of Gradle is the **task**:

```groovy
tasks.register('hello') {
    doLast {
        println 'Hello from Gradle!'
    }
}

tasks.register('count') {
    doLast {
        4.times { print "$it " }
    }
}
```

```bash
gradle hello         # Run the hello task
gradle build         # Build (compile + test + jar)
gradle clean         # Clean
gradle test          # Run tests
gradle bootJar       # Spring Boot packaging
gradle dependencies  # View dependency tree
```

Gradle's default tasks are far more flexible than Maven's — you can write actual code logic, such as dynamically altering dependencies based on environment.

### 2.3 Advantages

- **Concise DSL** — the same configuration is half the length of XML.
- **Fast builds** — incremental compilation, build cache, and a daemon process make large projects 2–10x faster than Maven.
- **Flexible** — Groovy/Kotlin lets you write arbitrary logic, freer than Maven's plugin mechanism.
- **Default for Android** — Android Studio projects are all Gradle.

## 3. Maven vs Gradle: How to Choose

| Aspect | Maven | Gradle |
| --- | --- | --- |
| Config file | `pom.xml` (XML) | `build.gradle` (Groovy/Kotlin DSL) |
| Learning curve | Gentle, strict conventions | Slightly steeper, flexible |
| Build speed | Moderate | **Fast** (incremental, cached) |
| Flexibility | Plugin mechanism, somewhat rigid | Arbitrary programming logic |
| Ecosystem | **Richest** | Rich (dominant in Android) |
| Multi-module | Supported but verbose | Elegant |
| Best for | Java backend, legacy projects | Android, new projects, large projects |

**Selection advice**:

- Company already on Maven → Maven (consistency matters most).
- Spring Boot projects → either works; follow team habit.
- Android projects → Gradle (mandatory).
- Large multi-module new projects → Gradle (faster builds).

## 4. Hands-On: Simulating Dependency Resolution

Since build-tool code can't run in Piston (it would require installing Maven/Gradle), below we use Java SE to simulate a **dependency resolver** — demonstrating the core logic of Maven's transitive dependencies and conflict resolution.

<CodeRunner code={`import java.util.*;
import java.util.stream.*;

// 模拟 Maven 依赖
class Dependency {
    final String group, artifact, version;
    final String scope;
    final List<Dependency> transitive;   // 传递依赖

    Dependency(String g, String a, String v, String scope, Dependency... transitive) {
        this.group = g; this.artifact = a; this.version = v;
        this.scope = scope;
        this.transitive = Arrays.asList(transitive);
    }
    String coordinate() { return group + ":" + artifact + ":" + version; }
    String key() { return group + ":" + artifact; }   // 不含版本, 用于冲突检测
    @Override public String toString() { return coordinate() + " (" + scope + ")"; }
}

// 简易依赖解析器
class DependencyResolver {
    // 模拟 Maven 仓库: 坐标 -> 依赖对象
    private final Map<String, Dependency> repo = new HashMap<>();

    void install(Dependency d) { repo.put(d.coordinate(), d); }

    // 解析项目的所有依赖 (含传递依赖, 解决冲突)
    public List<Dependency> resolve(List<String> directDeps) {
        // 最终结果: artifactKey -> Dependency (路径短/先声明的优先)
        Map<String, Dependency> resolved = new LinkedHashMap<>();
        Map<String, Integer> depths = new HashMap<>();

        for (String coord : directDeps) {
            Dependency d = repo.get(coord);
            if (d == null) {
                System.out.println("[警告] 找不到依赖: " + coord);
                continue;
            }
            resolveRecursive(d, 0, resolved, depths);
        }
        return new ArrayList<>(resolved.values());
    }

    private void resolveRecursive(Dependency dep, int depth,
            Map<String, Dependency> resolved, Map<String, Integer> depths) {
        String key = dep.key();
        Integer existDepth = depths.get(key);

        if (existDepth != null) {
            // 已有同 artifact 的依赖, 比较路径长度
            if (depth >= existDepth) {
                System.out.println("  [冲突] " + dep.coordinate()
                    + " (深度 " + depth + ") 被已有 "
                    + resolved.get(key).coordinate()
                    + " (深度 " + existDepth + ") 胜出, 跳过");
                return;
            }
            System.out.println("  [冲突] " + dep.coordinate()
                + " (深度 " + depth + ") 胜出已有 "
                + resolved.get(key).coordinate()
                + " (深度 " + existDepth + "), 替换");
        }

        System.out.println("  [解析] " + "  ".repeat(depth) + dep.coordinate());
        resolved.put(key, dep);
        depths.put(key, depth);

        // 递归处理传递依赖
        for (Dependency t : dep.transitive) {
            resolveRecursive(t, depth + 1, resolved, depths);
        }
    }
}

public class Main {

        DependencyResolver resolver = new DependencyResolver();

        // Spring-Web 依赖 Spring-Core 5.3.0
        // Spring-Web-MVC 也依赖 Spring-Core 5.3.1 (冲突!)
        // Commons-Lang3 是无传递依赖的工具库
        // JUnit 依赖 JUnit-Platform (test scope, 不会传递到主项目)
        Dependency springCore530 = new Dependency("org.springframework", "spring-core", "5.3.0", "compile");
        Dependency springCore531 = new Dependency("org.springframework", "spring-core", "5.3.1", "compile");
        Dependency commonsLang = new Dependency("org.apache.commons", "commons-lang3", "3.13", "compile");
        Dependency junitPlatform = new Dependency("org.junit.platform", "junit-platform", "1.10", "test");
        Dependency junit = new Dependency("org.junit.jupiter", "junit-jupiter", "5.10", "test", junitPlatform);

        Dependency springWeb = new Dependency("org.springframework", "spring-web", "5.3.0", "compile", springCore530);
        Dependency springMvc = new Dependency("org.springframework", "spring-webmvc", "5.3.1", "compile", springCore531);

> - **Coordinates uniquely identify a jar** — all three of `groupId:artifactId:version` are indispensable.

## 5. Chapter Summary

| Concept | Key Points |
| --- | --- |
| Build tool responsibilities | Dependency management, lifecycle, standardization, multi-module |
| Maven coordinates | `groupId:artifactId:version` |
| Maven scopes | compile/test/provided/runtime |
| Transitive dependencies | Automatically pull in dependencies of dependencies |
| Conflict resolution | Nearest path wins; same depth → first declared wins |
| Maven lifecycles | Three sets: clean / default / site |
| Maven repositories | Local → private → central |
| Gradle DSL | Groovy/Kotlin, concise and flexible |
| Gradle scopes | implementation/api/compileOnly/runtimeOnly |

**Mnemonic**:

- **Three coordinate elements** — groupId, artifactId, version.
- **Four scope tiers** — compile (default), test (testing), provided (provided by container), runtime (runtime only).
- **Conflicts: nearest path wins** — at the same depth, first declared wins.
- **Maven has three lifecycles** — clean, default, and site, independent of each other.
- **Gradle is faster than Maven** — incremental compilation + cache.
- **Android mandates Gradle** — otherwise follow the team.
